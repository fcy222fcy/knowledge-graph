# 一、产品概述

## 1.1  产品背景

大学生日常就餐频率高，选择集中在校内食堂档口、校门口小店、大学城商圈、外卖店铺和夜宵摊位。但相关餐饮信息分散在外卖平台、地图、微信群、朋友圈和同学口口相传中，评价可信度和检索效率较低。尤其是食堂档口、夜宵摊位等非标准店铺，往往缺少独立线上信息。食友旨在围绕大学城学生的真实就餐经验，建立一个可信、轻量、可持续沉淀的美食评价社区。

## 1.2  核心痛点

| 痛点 | 描述 |
| --- | --- |
| 校园餐饮信息分散 | 食堂档口、校门口小店、夜宵摊位和外卖店铺信息分布在不同平台和同学口口相传中，难以集中查询 |
| 评价不够可信 | 外卖平台评价可能存在刷单和营销，同学之间的真实体验更有参考价值，但缺少沉淀工具 |
| 非标准店铺缺少线上信息 | 食堂档口、临时摊位、夜市摊位往往没有独立 POI 或完整店铺页，难以被搜索和评价 |
| 选择效率低 | 大学生高频面临"今天吃什么""食堂哪家好吃""校门口哪家不踩雷"等日常决策 |
| 缺乏同校社交属性 | 吃饭是大学生活中的高频社交行为，现有平台缺少同校 / 同大学城学生之间的评价互动 |

## 1.3  产品定位

食友是一个面向大学城学生的真实美食评价与发现社区，覆盖校内食堂档口、校门口小店、大学城商圈餐饮、外卖店铺和夜宵摊位。平台通过学生真实评价、消费凭证验证、轻社交互动和店铺排行榜，帮助大学生更快决定"今天吃什么"。

## 1.4  版本范围与分期

本 PRD 描述食友完整产品方向， v1.0 MVP 仅交付 P0 范围。P1 / P2 为后续迭代规划，不纳入 v1.0 MVP 研发、测试和上线验收范围。

### P0：v1.0 MVP 必做

P0 目标是验证大学城学生是否愿意沉淀真实美食评价，并验证平台是否能承接食堂档口、夜宵摊位等非标准店铺数据。

P0 包含：

- 手机号注册与登录
- 游客浏览首页评价信息流
- 浏览店铺 / 档口详情
- 搜索店铺 / 档口和评价
- 标准店铺和非标准店铺展示
- 发布图文评价
- 多维度评分
- 图片上传
- 人均消费填写
- 消费凭证上传与人工审核
- 创建待确认店铺并发布评价
- 待确认店铺相似匹配、后台确认、合并、驳回和补充信息
- 评价点赞、踩和回复
- 评价举报
- 基础个人主页：头像、昵称、个人简介、我的评价
- 基础系统通知
- 管理员内容审核
- 管理员用户管理
- 管理员店铺管理

### P1：后续增强

P1 用于增强可信度、留存和社区关系，不纳入 v1.0 MVP 验收范围。

P1 包含：

- 校园身份认证
- 同校内容推荐和可信排序加权
- 收藏店铺 / 想去清单
- 关注用户
- 首页关注 Tab
- 关注列表和粉丝列表
- 用户贡献等级
- 店铺排行榜
- 排行榜算法优化
- 商家认领已有店铺
- 商家官方回复
- 商家维护店铺展示信息
- 推荐菜管理
- 用户信誉分细化规则
- 二次审核队列
- 账号注销

### P2：远期能力

P2 用于规模化运营、跨区域扩展和更复杂的社交能力，不纳入 v1.0 MVP 验收范围。

P2 包含：

- 新增店铺并认领
- 饭搭子 / 约饭功能：用户可围绕店铺或档口发起约饭邀请，其他用户可报名参与；该功能涉及陌生人社交、隐私安全、举报拉黑和沟通机制
- 数据看板
- 第三方地图 POI 批量导入
- 跨学校 / 跨大学城切换
- 高级搜索能力，包括拼音、错别字和复杂排序
- 自动化真实性识别
- 自动化内容审核
- 个性化推荐算法
- 小程序版本
- 跨城市扩张
- 复杂商家营销工具

# 二、用户角色与权限

## 2.1  角色定义

| 角色 | 定义 | 核心诉求 |
| --- | --- | --- |
| 游客 | 未注册或未登录的访问者 | 浏览大学城店铺 / 档口和评价，了解真实口碑 |
| 普通用户 | 完成手机号注册并登录的用户，可填写学校、校区等资料，未完成校园认证时不展示特殊身份标识 | 发布评价、参与互动、记录想去的店铺、建立个人美食口碑 |
| 学生用户 | 在普通用户基础上，完成校园身份认证的用户 | 在个人主页展示已认证学生身份，获得同校内容推荐、可信排序和评价可信度的适度加权 |
| 商家用户 | 通过认领审核、获得店铺管理权的用户，适用于校门口小店、大学城商圈店铺、食堂档口经营者等 | 维护店铺展示信息和推荐菜、回应用户评价 |
| 管理员 | 平台运营人员，负责当前大学城范围内的店铺和内容治理 | 审核内容、确认店铺、管理用户和商家认领申请 |

商家身份与普通用户身份并存于同一账号，普通用户通过申请认领店铺、经管理员审核通过后激活商家身份。激活商家身份后，该账号管理所认领店铺时以商家身份操作；发布普通评价时仍以普通用户身份操作，但不可对自己经营的店铺发布评价。若该账号已完成校园认证，前台仅在其个人主页展示"已认证学生"标识，评价列表和评价详情页默认不展示校园认证标识。

管理员账号独立存在，不在前台页面暴露入口。管理员账号在系统初始化时由研发通过后端脚本创建首个超级管理员账号；正式运营后，超级管理员可在后台「用户管理」页新增普通管理员账号并分配操作权限；前台不提供管理员注册或登录入口。

# 三、功能需求（前台）

## 需求 1：用户注册与登录（P0）

用户故事：作为一名新用户，我想通过手机号快速完成注册和登录，以便使用发布评价等需要身份验证的功能。

验收标准：

1. 系统提供手机号 + 短信验证码的统一入口：输入手机号并获取验证码，若该手机号未注册则自动完成注册并登录，若已注册则直接登录，不区分两个独立流程。

2. 短信验证码为 6 位纯数字，有效期 5 分钟，超时后需重新获取。

3. 同一手机号 60 秒内只能发送一次验证码，60 秒倒计时结束前"获取验证码"操作不可触发。

4. 短信验证码接口需具备风控限制：后端按手机号、IP 维度统计发送频次并执行拦截，具体阈值支持配置。触发风控后提示"操作过于频繁，请稍后再试"。

5. 验证码连续输入错误超过 5 次，锁定该手机号 15 分钟，期间无法登录，并提示"登录失败次数过多，请 15 分钟后再试"。

6. 登录成功后系统签发 JWT Token，有效期 7 天，过期后需重新登录。

7. 首次注册的用户，系统自动生成默认昵称（格式：食友 + 6 位随机字母数字组合），默认昵称不得包含手机号、学校、校区等个人身份信息；用户可在个人主页修改。

## 需求 1.1：校园身份认证（P1）

用户故事：作为一名大学城学生，我希望认证自己的校园身份，以便提升自己评价内容的可信度，并获得更符合本校生活场景的内容推荐。

验收标准：

8. 用户完成手机号注册后，可选择进行校园身份认证；未认证不影响浏览、发布评价、互动、收藏、关注等基础功能。

9. 校园认证支持学生证 / 校园卡照片认证和人工审核。

10. 用户提交校园认证时，必须选择学校和校区，并上传学生证、校园卡或其他可证明在校身份的材料。

11. 认证状态包括：未认证、认证中、已认证、认证失败。

12. 认证中状态下，用户仍可正常使用普通用户功能。

13. 认证通过后，用户个人主页展示"已认证学生"标识；评价列表和评价详情页默认不展示校园认证标识。

14. 未完成校园认证的普通用户不展示"未认证"或类似负向标识。

15. 认证失败时，系统通过站内通知告知失败原因，用户可重新提交认证。

16. 认证通过后，学校和校区字段不可自行修改；如需修改，用户需重新提交校园认证。

17. 学生证、校园卡等认证材料仅用于身份审核，前台不展示；平台需限制材料访问权限，仅管理员可在审核场景下查看。

18. 校园认证状态可用于"有帮"排序、同校内容推荐和评价可信度计算的适度加权，但该加权不直接改变单条评价评分，也不在评价卡片中单独展示校园认证标识。

19. 用户伪造认证材料或冒用他人身份时，管理员可撤销其校园认证状态，并视情况降低信誉分或封禁账号。

## 需求 1.2：账号注销（P1）

用户故事：作为一名注册用户，我希望能够注销自己的账号，以便在不再使用平台时删除个人可识别信息。

验收标准：

20. 登录用户可在"个人设置"页面发起账号注销申请；注销入口不在首页等高频页面直接暴露，需要用户主动进入设置页寻找。

21. 注销流程需通过手机号短信验证码二次确认用户身份，验证规则与登录验证码一致（6 位、5 分钟有效、同号 60 秒限频）。

22. 注销前系统向用户展示以下提示，需用户主动勾选确认后方可继续：

    - 账号注销后将无法恢复，所有个人资料（头像、昵称、简介、学校、关注关系、收藏记录）将被永久删除。
    - 您发布的评价内容将以匿名形式继续展示，昵称显示为"已注销用户"。
    - 已上传的消费凭证图片和校园认证材料将按隐私策略删除。
    - 若您持有商家认领资格，注销后该店铺将重新变为未认领状态，需另行申请认领。

23. 注销操作完成后立即生效：账号无法登录；个人资料字段（昵称、头像、简介、学校、校区）清空；评价内容以匿名形式继续展示，发布者昵称替换为"已注销用户"；该账号关联的消费凭证原始图片和校园认证材料按数据保留策略删除或不可逆脱敏；商家认领资格释放，店铺重新变为未认领状态。

24. 注销完成后，该手机号可重新注册新账号；新账号与原账号历史数据无关联。

25. 管理员后台不提供代用户注销的操作入口；管理员只能封禁账号，不能代替用户执行注销。

## 需求 1.3：微信公众号入口与移动端访问（P0）

用户故事：作为一名大学城学生，我希望可以通过微信公众号底部菜单直接进入食友移动端页面，以便在微信内快速浏览评价、搜索店铺、查看排行榜和管理个人主页。

验收标准：

26. 平台提供适配微信内置浏览器的 H5 移动端页面，支持从微信公众号底部菜单直接跳转访问。

27. 微信公众号底部菜单至少包含以下入口：首页 / 发现、排行榜、我的；分别跳转至移动端首页信息流、店铺排行榜页、个人主页或登录页。

28. 用户从微信公众号菜单进入页面时，系统应识别访问来源为微信内置浏览器，并按移动端布局展示页面。

29. 未登录用户访问需要登录的页面或功能时，应引导完成手机号验证码登录；登录成功后返回用户原先访问的页面。

30. 微信公众号菜单跳转地址必须使用 HTTPS 正式域名；正式上线前需完成微信公众号网页授权域名配置。

31. 移动端页面需适配主流手机屏幕宽度，不得出现横向滚动、按钮遮挡、内容被浏览器底部区域遮挡等问题。

## 需求 2：浏览店铺与搜索（P0）

用户故事：作为一名游客，我想浏览平台的美食评价并搜索附近的店铺，以便在点餐前了解真实口碑。

验收标准：

32. 首页以信息流形式展示平台内的评价帖子，默认按评价热度排序。评价热度由点赞数、踩数、回复数、发布时间、是否已验证等因素综合计算，用于内容发现，不等同于店铺排行榜排名。

33. 信息流支持按场景 / 品类筛选，筛选项包括：全部、食堂档口、校门口小店、外卖、奶茶咖啡、夜宵小摊、水果超市。

34. 信息流采用分页加载，每次加载 20 条，触底后自动加载下一页；加载完毕后提示"已经到底了"。

35. 搜索功能支持通过店铺 / 档口名称、所属场所、品类关键词、标签进行检索，搜索结果分"店铺 / 档口"和"评价"两类展示。搜索结果"店铺 / 档口"Tab 默认按「综合评分 × log(评价数量 + 1)」加权值降序排列，评价数量不足 3 条的店铺排在所有有效结果之后；搜索结果"评价"Tab 默认按发布时间倒序排列。

36. 搜索结果支持筛选：按场景 / 品类、按评分高低、按人均价格区间（≤20 元 / 21-50 元 / 51-100 元 / >100 元）、按大学城区域 / 校区范围。

37. 搜索关键词为空时不发起请求；搜索结果为空时提示"暂未找到相关内容，换个关键词试试"。

38. 店铺详情页按数据可用情况展示评价对象信息：基础信息包括名称、品类、地址 / 位置或所属场所。标准店铺展示地址 / 位置和导航入口；食堂档口、摊位等非标准店铺展示所属场所和位置说明，若所属场所有位置数据，则提供导航至所属场所。当评价数不足 3 条时，综合评分、人均消费展示为"数据积累中"；当评价数达到 3 条及以上时，展示综合评分、评价总数和人均消费；当评价数达到 5 条及以上时，系统可展示基于用户评价聚合生成的高频标签 / 推荐菜关键词。若店铺暂无评价，则展示"暂无评价，成为第一个分享体验的人"。若店铺已被商家认领，可展示商家维护的店铺简介、营业时间、联系电话、店铺封面图和推荐菜，并标注"商家提供"。

39. 店铺详情页评价列表默认按"综合"排序，优先展示已验证、有图、点赞多且时间较新的评价；用户可切换为"最新"、"最高评分"或"最低评分"排序。

40. 店铺评价列表支持按类型筛选：全部、仅看已验证、仅看有图。

41. 店铺详情页根据位置类型展示导航能力：标准店铺提供"去这里"入口，点击后跳转至高德地图或腾讯地图进行导航；非标准店铺优先导航至所属场所，若所属场所暂无位置数据，则展示"位置待补充"。

### "综合"排序算法

"综合"排序分值由以下因子计算，分值越高排名越靠前：

| 因子 | 计分规则 |
| --- | --- |
| 已验证消费 | +30 分 |
| 含图片 | +20 分 |
| 净点赞数（点赞 − 踩） | × 2 分，上限 +40 分；净点赞数为负则不加分 |
| 时间衰减系数 | 发布后 ≤7 天 × 1.0；8-30 天 × 0.85；31-90 天 × 0.70；>90 天 × 0.50；将上述三项之和乘以此系数 |
| 用户等级加权系数 | Lv.1 × 1.00；Lv.2 × 1.05；Lv.3 × 1.10；Lv.4 × 1.15；Lv.5 × 1.20；将时间衰减后的分值再乘以此系数 |

> 上述权重为初始配置值，产品和运营可在后台参数配置页调整，无需代码发布。

### 待确认店铺状态流转

待确认店铺从用户创建到管理员处理，可能进入以下状态；"待确认"仅表示店铺信息尚未被平台核验，不代表最终必须确认为正式店铺：

| 状态 | 触发条件 | 前台展示 | 参与排行榜 |
| --- | --- | --- | --- |
| 待确认 | 用户创建并提交 | 可展示，标注"店铺信息待确认" | ❌ |
| 需补充信息 | 管理员要求补充 | 可展示，标注"店铺信息待确认" | ❌ |
| 已确认（正式店铺） | 管理员确认 | 正常展示，移除待确认标注 | ✅（评价数 ≥ 5 后） |
| 已合并 | 管理员合并至已有店铺 | 不再展示；评价迁移至目标店铺；创建人及相关评价发布用户收到系统通知 | — |
| 已驳回 | 管理员驳回 | 不再展示；创建人及相关评价发布用户收到含原因的系统通知 | — |

### 店铺数据来源与展示原则

- 店铺基础信息可来自管理员导入、第三方地图 POI、用户创建的待确认店铺或商家新增并认领申请。

- 平台支持标准店铺和非标准店铺。非标准店铺包括食堂档口、场所内摊位、临时摊位等，不要求具备独立地图 POI 或独立经纬度。

- 店铺展示信息按数据可用情况渐进展示，不要求冷启动阶段字段完整。

- 商家维护的信息包括店铺简介、营业时间、联系电话、封面图和推荐菜，前台展示时需标注"商家提供"。

- 用户评价产生的数据包括评分、评价正文、图片、人均消费、消费凭证、标签和互动数据。

- 平台聚合数据包括综合评分、评价总数、人均消费、高频标签、推荐菜关键词、店铺热度指数和排行榜名次，由系统基于有效评价计算，不允许商家手动修改。

- 待确认店铺前台可展示详情和评价，但需在店铺名称附近标注"店铺信息待确认"，并提示用户"该店铺信息由用户创建，平台尚未确认"。

- 待确认店铺不参与排行榜、首页推荐聚合和店铺热度榜单；其评价可在用户主页和搜索结果中展示。

- 若待确认店铺被管理员确认为正式店铺，原有评价、图片、互动数据保留并迁移至正式店铺。

- 若待确认店铺与已有店铺重复，管理员应执行合并；合并后，其下评价、图片、互动数据迁移至目标店铺，原待确认店铺不再公开展示。

- 若待确认店铺明显无效、虚假、无法定位，或补充信息后仍无法确认，管理员可驳回；驳回后，该店铺及其下评价不再公开展示，并通过系统通知告知店铺创建人及相关评价发布用户。

## 需求 3：发布评价（P0）

用户故事：作为一名普通用户，我想对消费过的店铺发布图文评价和多维度评分，以便帮助其他用户做出更好的选择。

验收标准：

42. 只有登录后的普通用户才能发布评价，未登录用户触发发布操作时，提示引导登录。

43. 发布评价必须关联一个且仅一个主评价对象，评价对象可以是标准店铺，也可以是食堂档口、摊位等非标准店铺。用户发布评价时先搜索系统内已有店铺 / 档口；若未找到目标对象，可发起创建"待确认店铺"并继续发布评价。创建待确认店铺时，店铺 / 档口名称、位置类型和场景 / 品类为必填项。

44. 创建待确认店铺时，若位置类型为标准店铺，需至少填写地址描述或地图定位中的一项；若位置类型为食堂档口、场所内摊位等非标准店铺，需填写所属场所和位置说明；若位置类型为临时摊位，需填写出摊区域和位置说明。

45. 用户创建待确认店铺前，系统需根据店铺名称、所属场所、地址描述、位置说明、场景 / 品类进行相似店铺匹配，并展示可能相同的已有店铺或待确认店铺。

46. 若用户选择相似店铺中的某一项，则评价直接关联该店铺，不再创建新的待确认店铺。

47. 若用户确认均不是目标店铺，系统才允许继续创建待确认店铺；提交时需记录创建来源和创建人。

48. 同一用户 24 小时内最多创建 5 个待确认店铺；超过限制时提示"今日新增店铺数量已达上限，请明天再试"。

49. 评价内容、评分、图片、消费凭证和互动数据均归属于主评价对象；若用户想评价多个店铺或档口，需分别发布多条评价。待确认店铺可展示评价，但前台需标注"店铺信息待确认"，且不参与排行榜和平台推荐聚合；待管理员确认后转为正式店铺。

50. 评价正文为必填项，字数限制为 20 至 500 字，系统实时展示已输入字数及剩余字数。

51. 多维度评分为必填项，评分范围 1 至 5 分，支持 0.5 分精度。不同评价对象采用不同评分维度：外卖店铺为口味、性价比、包装、配送速度；堂食店铺 / 校门口小店为口味、性价比、环境、服务态度；食堂档口 / 夜宵摊位为口味、性价比、出餐速度、卫生感受。评分维度由所关联评价对象的类型自动决定。

52. 图片为选填项，最多上传 9 张，每张不超过 5MB，支持 JPG、PNG、WEBP 格式；不符合格式或大小要求时给出明确提示，不影响表单其他字段的填写。

53. 标签为选填项，用户从系统预设标签中多选，最多选 5 个。

54. 人均消费为选填项，只允许输入正整数，单位为元。

55. 消费凭证为选填项，用户可上传外卖订单截图、小票照片或支付记录截图。上传前系统需提示用户遮挡手机号、姓名、宿舍地址、详细配送地址、订单号、支付账号、二维码和条形码等敏感信息。上传后该评价的验证状态标记为"待验证"，评价正文仍按正常规则公开展示；审核通过后展示"已验证消费"标签，审核拒绝后不展示负向标识，并通过站内通知告知用户拒绝原因。

56. 同一用户对同一店铺 30 天内只能发布 1 条评价；尝试重复发布时提示"您在 30 天内已对该店铺发表过评价"，禁止提交。

57. 商家不可对自己认领的店铺发布评价，系统在关联店铺时进行检测，检测到时提示"您是该店铺的认领商家，不可对其发布评价"。

58. 用户可编辑自己发布的评价。评价被编辑后，页面展示"已编辑"标识，并记录最后编辑时间。若修改评分、图片或消费凭证，评价进入待审核 / 待验证状态；审核或验证通过后重新公开对应内容。已被管理员隐藏、举报成立或处于审核处理中的评价不可编辑。

59. 用户可删除自己发布的任意评价，删除后该评价从所有列表中移除，相关互动数据同步删除。

## 需求 4：评价互动（P0）

用户故事：作为一名普通用户，我想对他人的评价进行点赞、踩或回复，以便表达看法并与其他用户交流。

验收标准：

60. 只有登录后的普通用户才能进行点赞、踩、回复操作，游客触发时提示引导登录。

61. 点赞与踩为互斥操作：同一用户对同一条评价只能选择点赞或踩其中之一。已点赞后再次点击为取消点赞；已踩后再次点击为取消踩；已点赞后点击踩，则自动取消点赞并记录踩，反之亦然。

62. 点赞数与踩数在操作后实时更新并展示。

63. 回复支持直接回复评价本身，也支持回复他人的回复，最多 2 层嵌套；第 2 层的回复不再继续嵌套，统一展示在第 1 层回复下方。

64. 单条回复字数限制为 1 至 200 字。

65. 用户可删除自己发布的回复；删除后，该回复的子回复同步删除。

66. 商家可以商家身份对自己店铺下的评价进行官方回复，官方回复带有"商家回复"标识，展示位置置顶于该评价所有回复之上；每条评价只允许一条商家官方回复，已回复后可修改内容，但不可再新增。

## 需求 5：收藏与关注（P1）

用户故事：作为一名普通用户，我想收藏感兴趣的店铺并关注优质用户，以便日后快速找到并获取他们的新内容。

验收标准：

67. 普通用户可收藏任意店铺，收藏后保存在个人主页的"想去清单"中；再次点击收藏则取消收藏。

68. 普通用户可关注其他普通用户或校园认证用户，关注关系为单向；再次点击关注则取消关注；用户不可关注自己。

69. 首页信息流提供"关注"Tab，切换后仅展示已关注用户发布的最新评价，按发布时间倒序排列。

70. 个人主页展示该用户的关注数与粉丝数，点击可分别查看关注列表和粉丝列表。

## 需求 6：个人主页（P0 / P1）

用户故事：作为一名普通用户，我想管理自己的个人资料并查看发布过的评价，以便建立个人美食口碑形象。

验收标准：

71. 每位普通用户均有独立的个人主页，所有人均可查看他人的公开主页。

72. 个人主页展示：头像、昵称、个人简介、所在城市、加入时间、用户贡献等级、发布评价数、获赞总数、关注数、粉丝数。

73. 平台建立用户贡献等级体系，前台展示用户等级标识。等级包括：Lv.1 新食友、Lv.2 探店新手、Lv.3 活跃食友、Lv.4 优质食友、Lv.5 可信达人。用户等级由贡献值计算，贡献值来源包括账号存续时间、有效评价数量、已验证评价数量、有图评价数量、评价互动反馈和违规记录。

74. 贡献值规则：账号注册满 7 天 +5，注册满 30 天 +15；发布公开评价 +5，发布已验证评价 +10，发布有图评价 +3；评价正文超过 80 字且未被举报成立 +2；评价获得点赞 +1，每条评价最多计 20 分；评价获得收藏 +2，每条评价最多计 20 分；评价被举报成立或被管理员隐藏 −20。

75. 等级门槛：Lv.1 新食友 = 0 分，Lv.2 探店新手 ≥ 30 分，Lv.3 活跃食友 ≥ 100 分，Lv.4 优质食友 ≥ 250 分，Lv.5 可信达人 ≥ 500 分且已验证评价不少于 10 条、举报成立次数为 0。

76. 用户等级主要用于身份展示、评价列表排序辅助和审核策略优化，不直接修改店铺综合评分和排行榜评分。高等级用户的评价在"有帮助"排序中可获得轻微加权；低信誉用户发布的新评价进入审核队列。

77. 用户只能编辑自己的个人资料，可修改字段为：头像、昵称、个人简介、所在城市。昵称长度限制为 2 至 20 个字符，不允许包含特殊符号。

78. 个人主页包含以下内容 Tab：我的评价、想去清单、关注、粉丝。若当前用户已激活商家身份，个人主页展示"我的店铺"入口，点击后进入商家店铺管理页面。

79. 本人查看自己的评价列表时，符合编辑条件的评价展示编辑入口，任意时间展示删除入口；他人查看时不展示任何操作入口。

## 需求 7：店铺排行榜（P1）

用户故事：作为一名用户，我想查看按真实口碑排名的店铺榜单，以便快速发现当前最受好评的餐厅。

验收标准：

80. 店铺排行榜用于展示当前大学城范围内的店铺 / 档口排名，不展示评价帖子排名。排行榜按大学城学生高频消费场景设立榜单，包括：大学城综合榜、食堂档口榜、校门口小店榜、外卖榜、奶茶咖啡榜、夜宵小摊榜、水果超市榜。

81. 每个榜单最多展示 Top 20，评价数量不足 5 条的店铺不参与排行，显示为"评价积累中"。

82. 公开状态的有效评价参与店铺综合评分、店铺热度指数和排行榜计算；待审核、隐藏、删除、举报成立的评价不参与计算。待确认店铺及其评价不参与店铺排行榜计算，待店铺转为正式状态且评价数量达到 5 条后再纳入计算。

83. 店铺综合评分由该店铺下所有有效评价汇总生成。单条评价综合评分 = 口味评分 × 50% + 性价比评分 × 20% + 第三维度评分 × 15% + 第四维度评分 × 15%；店铺综合评分取有效评价的单条综合评分平均值。若某店铺差评（2 分及以下）占比超过 40%，店铺综合评分乘以惩罚系数 0.85。

84. 店铺热度指数仅统计近 7 天数据，由该店铺下公开评价的互动数据和店铺收藏数据计算：评价点赞数 − 评价踩数 × 1.5 + 评价回复数 × 0.3 + 店铺收藏数 × 0.5。

85. 店铺最终排名权重：店铺综合评分 × 50% + 评价可信度 × 25% + 店铺热度指数（归一化后）× 15% + 已验证评价占比 × 10%。评价可信度由有效评价数量、评分分布、差评比例和异常评分情况综合计算。

86. 排行榜数据每日凌晨 02:00 全量刷新一次，页面顶部展示最近一次更新时间。

87. 排行榜默认展示当前大学城范围内榜单；用户可切换学校、校区或大学城区域。

88. 排行榜支持按人均价格区间筛选：≤20 元 / 21-50 元 / 51-100 元 / >100 元。

## 需求 8：消费真实性验证（P0）

用户故事：作为一名普通用户，我想通过上传消费凭证为评价增加可信度，以便让其他用户相信我的评价是真实的。

验收标准：

89. 用户发布评价时可选择上传消费凭证，支持外卖订单截图、小票照片、支付记录截图等能证明消费行为的材料。

90. 上传消费凭证前，系统需提示用户遮挡个人敏感信息，包括但不限于手机号、姓名、宿舍地址、详细配送地址、订单号、支付账号、二维码和条形码。

91. 有效消费凭证需至少包含以下信息中的两类：商家或档口名称、消费时间、消费金额、商品信息，并能与所评价店铺或档口基本匹配。

92. 消费凭证为选填项；未上传消费凭证的评价正常公开展示，不展示"未验证"或类似负向标识。

93. 用户上传消费凭证后，该评价的验证状态变更为"待验证"；评价正文仍可按正常规则公开展示，但不展示"已验证消费"标签。

94. "待验证"评价由管理员在后台进行人工审核。审核通过后，验证状态变更为"已验证"，评价列表和详情页展示"已验证消费"标签；审核拒绝后，验证状态变更为"验证失败"，不展示负向标识，并通过站内通知告知用户拒绝原因。

95. 消费凭证图片仅用于平台审核，前台不展示，其他用户、商家用户均不可查看。

96. 管理员仅可在凭证审核场景下查看消费凭证图片；查看、通过、拒绝操作均需记录操作日志。

97. 平台应限制消费凭证图片的访问权限，图片 URL 不应可被未授权访问或长期公开访问。

98. 消费凭证审核完成后，平台保留审核结论、审核时间、审核人和拒绝原因；原始凭证图片最多保留 30 天，到期后自动删除或转为不可逆脱敏存档。

99. 用户删除评价后，关联消费凭证图片不再用于审核和展示，并按平台数据保留策略删除或脱敏。

100. 用户注销账号后，其消费凭证图片应按隐私策略删除或脱敏处理；匿名保留的评价内容不得继续关联可识别用户身份的凭证材料。

101. 同一用户 30 天内消费凭证被拒绝累计超过 3 次，该用户新发布的评价自动进入二次审核队列，不立即公开展示，由管理员审核通过后再公开。

102. 用户信誉分为内部风控指标，对外不展示具体分值；贡献等级为前台展示指标。信誉分初始 100 分，消费凭证审核通过 +5 分，评价被举报成立 −10 分，30 天内重复评价被系统拦截 −3 分，消费凭证多次被拒绝 −5 分；信誉分低于 60 分时，该用户新发布的评价自动进入审核队列，不立即公开。信誉分低于 40 分时，系统限制其发布频率。

103. 消费凭证审核应在提交后 48 小时内处理完成；超过 48 小时未处理时，后台列表标记为"超时待处理"。

## 需求 9：举报机制（P0）

用户故事：作为一名普通用户，我想举报我认为虚假或违规的评价，以便帮助平台维护内容质量。

验收标准：

104. 普通用户可对任意评价发起举报，举报时必须选择原因：内容虚假或水军、广告推广、低俗内容、人身攻击、其他。选择"其他"时需填写补充说明，字数限制为 5 至 200 字。

105. 同一用户对同一条评价只能举报一次，重复举报时提示"您已举报过该评价"。

106. 举报提交后进入管理员待处理队列；处理完成后向举报人推送系统通知"您的举报已处理完毕"。

107. 若同一条评价被 3 名及以上不同用户举报，系统自动将该评价置为隐藏状态，从公开列表中移除，等待管理员处理。

108. 管理员处理结果为"成立"或"驳回"。若判定成立，评价永久下架，被举报用户信誉分 −10；若判定驳回，举报记录标记为已驳回。若同一条评价因达到 3 次举报阈值被自动隐藏，管理员须将该评价下所有举报记录全部驳回后，评价才自动恢复公开展示；只要有任意一条举报被判定为成立，该评价即永久下架，不可恢复。

## 需求 10：商家店铺管理（P1 / P2）

用户故事：作为一名商家，我想认领并管理自己的店铺主页，以便向用户展示准确的店铺信息和推荐菜并及时回应评价。

验收标准：

109. 普通用户可在店铺详情页发起认领申请，填写申请人姓名、联系电话、与店铺的关系，并可上传营业执照等证明材料。

110. 商家搜索不到自己的店铺时，可提交"新增店铺并认领"申请，填写店铺名称、所属类型、经营品类、地址描述、联系电话、营业时间，并上传营业执照或其他证明材料。管理员审核通过后，系统创建正式店铺并将该店铺绑定至该商家账号；审核拒绝时需说明原因。

111. 认领申请提交后由管理员审核：通过后申请人获得该店铺的商家管理权限；拒绝时系统通知申请人并说明拒绝原因。

112. 同一店铺同一时间只能有一位商家认领；已被认领的店铺，认领入口不展示，显示"该店铺已由商家认领"。

113. 商家可在"我的店铺"中维护店铺展示信息，包括店铺简介、营业时间、人均消费参考、联系电话、店铺封面图（最多 5 张）。店铺名称、地址、经纬度由管理员维护，商家不可自行修改；商家维护的信息在前台展示时需标注"商家提供"。

114. 商家可在"我的店铺"中新增、编辑、删除推荐菜信息。每道推荐菜包含：菜品名称（必填，最多 30 字）、菜品图片（选填，最多 3 张）、价格（选填）、菜品描述（选填，最多 100 字）、展示排序；商家维护的推荐菜在前台展示时需标注"商家提供"。

115. 商家可对自己店铺下的评价进行一条官方回复，已回复后可修改，但不可新增第二条；商家不可删除、举报或修改用户发布的评价。

# 四、功能需求（后台管理）

## 需求 11：管理员 — 内容审核（P0 / P1）

用户故事：作为管理员，我想在后台集中处理消费凭证审核和举报投诉，以便维护平台内容的真实性与社区秩序。

验收标准：

116. 后台提供"待审核凭证"列表，展示所有验证状态为"待验证"的评价，列表字段包含：提交人昵称、关联店铺名称、评价摘要、凭证图片、提交时间、是否超时。

117. 管理员可在列表页预览凭证图片；系统仅允许具备内容审核权限的管理员查看凭证图片，并记录查看日志。

118. 管理员对每条待审核凭证执行通过或拒绝操作；拒绝时必须填写拒绝原因，该原因将通过系统通知发送给用户；操作完成后该记录从待审核列表移除，并记录审核人、审核时间和审核结论。

119. 超过 48 小时未处理的凭证审核记录，在列表中标记为"超时待处理"并置顶展示。

120. 后台提供"待处理举报"列表，列表字段包含：举报人昵称、被举报评价内容摘要、举报原因、举报时间、该评价已被举报次数。

121. 因达到 3 次举报阈值被自动隐藏的评价，在列表中置顶并标注"已自动隐藏"，供管理员优先处理。

122. 管理员对每条举报执行"成立（下架评价）"或"驳回"操作，操作完成后该举报记录从待处理列表移除。

123. 后台提供"二次审核队列"，展示因用户信誉分低于 60 分而被拦截的评价，管理员可通过或拒绝发布；拒绝时向用户发送系统通知。

124. 用户编辑评价后，若修改评分、图片或消费凭证，系统将该评价重新进入待审核 / 待验证流程；管理员可查看编辑前后差异，并决定通过、驳回或隐藏。

## 需求 12：管理员 — 用户管理（P0 / P1）

用户故事：作为管理员，我想管理平台的用户信息，以便处理违规账号，维护平台正常运营。

验收标准：

125. 后台提供用户列表，列表字段包含：用户 ID、昵称、手机号、注册时间、评价数量、贡献等级、信誉分、账号状态（正常 / 封禁）。

126. 用户列表支持按昵称、手机号、账号状态筛选。

127. 管理员可对指定用户执行封禁操作，封禁时需填写封禁时长（1 天 / 7 天 / 30 天 / 永久）和封禁原因；被封禁用户在封禁期内无法登录，尝试登录时提示"您的账号已被封禁，原因：xxx，封禁至：xxx"。

128. 管理员可对已封禁用户执行解封操作，解封后用户可立即重新登录。

129. 管理员可手动调整用户信誉分，调整时需填写原因，操作记录保存至操作日志中。

130. 后台提供商家认领申请列表，支持处理"认领已有店铺"和"新增店铺并认领"两类申请；列表字段包含：申请人昵称、目标店铺名称或拟新增店铺名称、申请时间、提交的证明材料。管理员执行通过或拒绝操作，拒绝时需填写原因。

131. 后台提供校园认证申请列表，列表字段包含：申请人昵称、学校、校区、认证材料、申请时间、当前状态。管理员可预览认证材料，并执行通过或拒绝操作；拒绝时必须填写原因，该原因将通过系统通知发送给用户。认证通过后，用户状态变更为校园认证用户。

## 需求 13：管理员 — 店铺管理（P0 / P2）

用户故事：作为管理员，我想维护平台的店铺基础数据，以便保证用户能搜索到准确完整的店铺信息。

验收标准：

132. 后台提供店铺列表，支持按店铺名称、场景 / 品类、位置类型、大学城区域 / 校区范围、是否已认领、确认状态进行筛选。

133. 管理员可新增正式店铺。标准店铺必填字段为：店铺名称、所属类型（外卖 / 实体店）、经营品类、经纬度；选填字段为：地址描述、营业时间、联系电话。非标准店铺必填字段为：名称、位置类型、所属场所或位置说明；可不填写独立经纬度。

134. 后台提供待确认店铺列表，列表字段包括：店铺名称、位置类型、所属场所、位置说明、地址描述、场景 / 品类、创建人、关联评价数、创建时间、相似店铺提示、当前状态。

135. 管理员需处理用户创建的待确认店铺，处理方式包括确认为正式店铺、合并至已有店铺、要求补充信息或驳回；待确认店铺不要求最终必须被确认为正式店铺。

136. 确认为正式店铺时，管理员需补全店铺信息，设置位置类型、所属场所、地址或位置说明后，店铺确认状态变更为已确认，原有评价、图片和互动数据保留。

137. 合并至已有店铺时，管理员需选择目标店铺并填写合并原因；系统将待确认店铺下的评价、图片和互动数据迁移至目标店铺，原待确认店铺不再公开展示，并生成店铺合并记录；同时通知原待确认店铺创建人及相关评价发布用户，通知内容包含目标店铺和合并原因。

138. 驳回待确认店铺时，适用于明显无效、虚假、无法定位，或补充信息后仍无法确认的对象；重复店铺应优先执行合并。驳回时管理员需填写驳回原因；驳回后该店铺及其下评价不再公开展示，并通知店铺创建人及相关评价发布用户。

139. 要求补充信息时，管理员需选择需补充的信息项并填写说明；系统记录需补充的信息项和补充说明，并通知创建人补充所属场所、地址描述或位置说明，补充完成后重新进入待确认店铺列表。

140. 管理员可编辑任意店铺的所有字段，包括商家不可自行修改的店铺名称和地址。

141. 管理员可下线店铺，下线后该店铺不再出现在搜索结果和排行榜中，但历史评价数据保留；已下线店铺可重新上线。

142. 管理员可撤销商家对某店铺的认领权限，撤销后商家失去该店铺管理权限，店铺重新变为未认领状态。

143. 管理员可通过上传 CSV 文件或第三方地图 POI 服务批量导入当前大学城范围内的基础店铺数据；导入时系统校验必填字段，校验失败的行以错误报告形式返回，不影响其余行的正常导入。

# 六、非功能需求

## 6.1  性能要求

| 指标 | 要求 | 备注 |
| --- | --- | --- |
| 首页信息流首次加载 | ≤ 2 秒 | 4G 网络环境下 |
| 搜索响应时间 | ≤ 1.5 秒 | 从提交请求到返回结果 |
| 图片上传 | ≤ 8 秒 | 5MB 图片，正常网络环境 |
| 排行榜加载 | ≤ 1 秒 | 数据已预计算，直接读缓存 |
| 并发用户数 | 支持 500 并发 | MVP 阶段目标 |

## 6.2  安全要求

- 所有需要用户身份的接口必须校验 JWT Token，Token 无效或过期时返回 401 状态码。

- 用户密码（管理员账号密码）必须使用 bcrypt 加密存储，禁止明文存储。

- 图片上传接口须校验文件 MIME Type，拒绝非图片格式文件。

- 所有接口须对用户输入进行严格校验，防止 SQL 注入和 XSS 攻击。

- 短信验证码接口须做频率限制，防止短信轰炸；频率规则至少包含单号 60 秒 1 次，并支持按手机号、IP 配置每日发送上限。

- 管理员后台接口须独立鉴权，与前台用户接口分离，不在前台路由中暴露。

- 平台初始店铺数据仅使用合法授权的数据来源，包括管理员录入、商家提交、用户创建和第三方地图 POI 服务。初始数据范围限定为当前大学城及周边学生高频消费区域。未经授权，不抓取或复制第三方外卖平台的评分、评论、图片、菜单、榜单、推荐菜等内容。

## 6.3  兼容性要求

- PC 端支持 Chrome、Edge、Safari 最新两个大版本。

- 移动端页面需在 iOS 15+ 和 Android 10+ 系统的主流浏览器中正常使用。

- 图片上传支持格式：JPG、JPEG、PNG、WEBP。

## 6.4  数据要求

- 用户发布的评价数据永久保留；用户注销账号后，评价内容以匿名形式继续展示，昵称显示为"已注销用户"。

- 用户上传的消费凭证图片仅用于真实性审核，前台不展示，其他用户和商家不可查看。

- 消费凭证审核完成后，平台保留审核结论、审核时间、审核人和拒绝原因；原始凭证图片最多保留 30 天，到期后自动删除或转为不可逆脱敏存档。

- 用户删除评价或注销账号后，关联消费凭证图片应按隐私策略删除或脱敏处理，不得继续关联可识别用户身份的信息。

- 用户上传的校园认证材料仅用于身份审核，前台不展示；管理员仅可在审核场景中查看，平台应记录查看和处理日志。

- 管理员的所有后台操作日志保留 180 天。

## 6.5  系统通知

- 以下事件须向用户发送站内系统通知：消费凭证审核通过、消费凭证审核拒绝（含原因）、校园认证审核通过、校园认证审核拒绝（含原因）、举报处理完毕、评价被管理员下架、待确认店铺要求补充信息（含补充项和说明）、待确认店铺被合并（含目标店铺和合并原因）、待确认店铺被驳回（含驳回原因）、账号被封禁（含原因和截止时间）、账号注销成功。

- 系统通知以站内消息形式实现，不发送短信或 App 推送。

## 6.6  异常处理要求

以下为各类用户可见异常的兜底处理规范，开发须保证在上述场景下均有符合预期的用户提示，不允许白屏、无响应或展示技术性报错信息。

| 场景 | 兜底行为 |
| --- | --- |
| 图片上传失败（网络中断 / 服务端错误） | 提示"图片上传失败，请检查网络后重试"；已填写的表单其他字段不清空 |
| 图片格式或大小不符合要求 | 提示"不支持该格式 / 图片大小超过 5MB，请重新选择"；仅拦截不符合要求的图片，其他已上传图片保留 |
| 短信发送失败（运营商侧失败） | 提示"短信发送失败，请稍后重试或更换手机号"；不扣减用户发送次数配额 |
| 接口请求超时（>10 秒无响应） | 提示"网络请求超时，请检查网络连接后重试"；提供"重试"按钮 |
| 评价提交失败（服务端 5xx） | 提示"提交失败，请稍后重试"；保留用户已输入的评价内容，不清空表单 |
| 搜索服务不可用 | 提示"搜索服务暂时不可用，请稍后再试"；不展示空列表 |
| 信息流加载失败 | 提示"加载失败，点击刷新"；展示刷新入口 |
| 登录 Token 过期 | 静默跳转至登录页，同时在页面顶部展示"登录已过期，请重新登录"提示 |
| 消费凭证上传失败 | 提示"凭证上传失败，请重试；评价其他内容不受影响，可先提交评价，稍后在编辑中补传凭证" |
| 操作触发风控拦截 | 提示"操作过于频繁，请稍后再试"；不暴露具体风控规则和阈值 |
