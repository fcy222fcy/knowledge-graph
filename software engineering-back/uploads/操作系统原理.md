# 操作系统原理详解

## 一、操作系统概述

### 1.1 什么是操作系统

操作系统（Operating System, OS）是管理计算机硬件与软件资源的系统软件，是用户与计算机之间的接口。操作系统的主要功能包括：

- **处理器管理**：进程调度、中断处理、多任务
- **存储器管理**：内存分配与回收、虚拟内存
- **设备管理**：I/O 设备分配与控制、驱动程序
- **文件管理**：文件存储、检索、共享、保护
- **用户接口**：命令行接口（CLI）、图形用户界面（GUI）、程序接口（API）

### 1.2 操作系统分类

| 类型 | 特点 | 代表 |
|------|------|------|
| 批处理系统 | 无交互，成批处理作业 | 早期大型机 |
| 分时系统 | 多用户同时交互 | UNIX、Linux |
| 实时系统 | 响应时间确定，高可靠 | VxWorks、FreeRTOS |
| 分布式系统 | 多机协作，透明性 | Hadoop、Spark |
| 嵌入式系统 | 资源受限，专用 | Android、嵌入式 Linux |

### 1.3 操作系统结构

```
┌─────────────────────────────────────┐
│           用户应用程序               │
├─────────────────────────────────────┤
│        系统调用接口（System Call）    │
├─────────────────────────────────────┤
│              内核                   │
│  ┌─────────┬─────────┬──────────┐  │
│  │进程管理  │内存管理  │文件系统   │  │
│  ├─────────┼─────────┼──────────┤  │
│  │设备驱动  │网络协议栈 │安全模块   │  │
│  └─────────┴─────────┴──────────┘  │
├─────────────────────────────────────┤
│           硬件抽象层（HAL）          │
├─────────────────────────────────────┤
│           计算机硬件                │

```

## 二、进程管理

### 2.1 进程与线程

**进程（Process）**：程序的一次执行实例，拥有独立的地址空间。

**线程（Thread）**：进程内的执行单元，共享进程的地址空间。

| 比较项 | 进程 | 线程 |
|--------|------|------|
| 地址空间 | 独立 | 共享 |
| 创建开销 | 大 | 小 |
| 切换开销 | 大（需切换页表） | 小（同一地址空间） |
| 通信方式 | IPC（管道、共享内存等） | 直接读写共享变量 |
| 容错性 | 一个进程崩溃不影响其他 | 一个线程崩溃整个进程终止 |

### 2.2 进程状态

```
                    创建
                     │
                     ▼
            ┌───────就绪◄──────────┐
            │        │             │
            │        ▼             │
            │      运行 ──→ 终止    │
            │        │             │
            │        ▼             │
            └──→ 阻塞 ────────────┘
                  (等待I/O)
```

**五状态模型详解**：

| 状态 | 说明 |
|------|------|
| 新建（New） | 进程正在被创建 |
| 就绪（Ready） | 进程已获得除CPU外的所有资源，等待调度 |
| 运行（Running） | 进程正在CPU上执行 |
| 阻塞（Blocked） | 进程等待某事件（如I/O完成）而暂停执行 |
| 终止（Terminated） | 进程执行完毕或被强制终止 |

### 2.3 进程调度算法

#### FCFS（先来先服务）
```
进程   到达时间   执行时间   完成时间   周转时间   带权周转
P1     0         24        24        24        1.0
P2     1         3         27        26        8.67
P3     2         3         30        28        9.33
P4     3         4         34        31        7.75
```

特点：简单公平，但对短作业不利（护航效应）。

#### SJF（短作业优先）
优先选择执行时间最短的进程。分为：
- **非抢占式**：一旦开始执行就不会被打断
- **抢占式（SRTF）**：新到达的进程如果执行时间更短，可以抢占当前进程

#### 时间片轮转（Round Robin）
每个进程获得一个固定的时间片（如20ms），时间片用完则切换到下一个进程。

时间片选择的影响：
- 太大 → 退化为 FCFS
- 太小 → 上下文切换开销过大
- 通常选择 10-100ms

#### 多级反馈队列（MLFQ）
```
优先级队列1（时间片=8ms）  ← 新进程进入
优先级队列2（时间片=16ms）
优先级队列3（时间片=32ms）
优先级队列4（FCFS）
```

规则：
1. 新进程进入队列1
2. 在当前队列时间片内未完成，降级到下一队列
3. 调度时优先调度高优先级队列
4. 低优先级队列的进程长时间等待后可提升优先级

### 2.4 进程同步

#### 互斥与同步

**互斥（Mutual Exclusion）**：多个进程不能同时访问临界资源。

**同步（Synchronization）**：进程间协作，按一定顺序执行。

#### 信号量（Semaphore）

```c
// 信号量定义
typedef struct {
    int value;           // 信号量值
    struct proc *list;   // 等待队列
} semaphore;

// P操作（等待/减）
void P(semaphore *S) {
    S->value--;
    if (S->value < 0) {
        // 将当前进程加入等待队列
        block(S->list);
    }
}

// V操作（释放/加）
void V(semaphore *S) {
    S->value++;
    if (S->value <= 0) {
        // 唤醒等待队列中的一个进程
        wakeup(S->list);
    }
}
```

#### 经典同步问题

**生产者-消费者问题**：
```c
semaphore mutex = 1;    // 互斥访问缓冲区
semaphore empty = n;    // 空缓冲区数量
semaphore full = 0;     // 满缓冲区数量

// 生产者
void producer() {
    while (1) {
        produce_item();
        P(&empty);      // 等待空位
        P(&mutex);      // 进入临界区
        put_item();
        V(&mutex);      // 离开临界区
        V(&full);       // 增加满位
    }
}

// 消费者
void consumer() {
    while (1) {
        P(&full);       // 等待数据
        P(&mutex);      // 进入临界区
        get_item();
        V(&mutex);      // 离开临界区
        V(&empty);      // 增加空位
        consume_item();
    }
}
```

**哲学家就餐问题**：
```c
// 解决方案：限制同时就餐的哲学家数量
semaphore fork[5] = {1, 1, 1, 1, 1};
semaphore limit = 4;  // 最多4人同时拿筷子

void philosopher(int i) {
    while (1) {
        think();
        P(&limit);         // 限制就餐人数
        P(&fork[i]);       // 拿左筷子
        P(&fork[(i+1)%5]); // 拿右筷子
        eat();
        V(&fork[(i+1)%5]); // 放右筷子
        V(&fork[i]);       // 放左筷子
        V(&limit);
    }
}
```

### 2.5 死锁

#### 死锁的四个必要条件

1. **互斥条件**：资源一次只能被一个进程使用
2. **请求与保持条件**：进程持有资源的同时请求新资源
3. **不可剥夺条件**：已分配的资源不能被强制回收
4. **循环等待条件**：存在进程间的循环等待链

#### 死锁处理策略

| 策略 | 说明 | 优缺点 |
|------|------|--------|
| 预防 | 破坏四个条件之一 | 限制严格，资源利用率低 |
| 避免 | 动态检查是否安全（银行家算法） | 需要预知最大需求 |
| 检测+恢复 | 允许死锁发生，检测后恢复 | 开销大 |
| 忽略 | 不处理（多数OS采用） | 简单但危险 |

#### 银行家算法

```
Available（可用资源）: [3, 3, 2]
Max（最大需求矩阵）:
  P0: [7, 5, 3]
  P1: [3, 2, 2]
  P2: [9, 0, 2]
  P3: [2, 2, 2]
  P4: [4, 3, 3]

Allocation（已分配矩阵）:
  P0: [0, 1, 0]
  P1: [2, 0, 0]
  P2: [3, 0, 2]
  P3: [2, 1, 1]
  P4: [0, 0, 2]

Need（还需矩阵）= Max - Allocation:
  P0: [7, 4, 3]
  P1: [1, 2, 2]
  P2: [6, 0, 0]
  P3: [0, 1, 1]
  P4: [4, 3, 1]

安全序列: P1 → P3 → P4 → P0 → P2
```

## 三、内存管理

### 3.1 内存分配方式

#### 连续分配

| 方式 | 说明 | 问题 |
|------|------|------|
| 单一连续分配 | 只有一道用户程序 | 内存利用率低 |
| 固定分区分配 | 分为固定大小的区域 | 内碎片 |
| 动态分区分配 | 按需分配 | 外碎片 |

#### 动态分区分配算法

| 算法 | 策略 | 特点 |
|------|------|------|
| 首次适应（FF） | 从头开始找第一个够大的空闲区 | 简单，空闲区在前面聚集 |
| 最佳适应（BF） | 找最小的够大空闲区 | 产生大量小碎片 |
| 最差适应（WF） | 找最大的空闲区 | 大空闲区很快被分割 |
| 下次适应（NF） | 从上次位置开始找 | 分布均匀 |

### 3.2 分页存储

将逻辑地址空间和物理内存都分成固定大小的块：

```
逻辑地址空间          物理内存
┌─────┐              ┌─────┐
│ 页0 │ ──────────→  │ 帧3 │
├─────┤              ├─────┤
│ 页1 │ ──────────→  │ 帧1 │
├─────┤              ├─────┤
│ 页2 │ ──────────→  │ 帧7 │
├─────┤              ├─────┤
│ 页3 │ ──────────→  │ 帧5 │
└─────┘              └─────┘

页表: 页号 → 帧号
0 → 3
1 → 1
2 → 7
3 → 5
```

**地址转换**：
```
逻辑地址 = 页号 + 页内偏移
例: 逻辑地址 = 0x1A3F (页大小=4KB=0x1000)
    页号 = 0x1A3F / 0x1000 = 1
    页内偏移 = 0x1A3F % 0x1000 = 0x0A3F
    查页表: 页1 → 帧1
    物理地址 = 1 × 0x1000 + 0x0A3F = 0x1A3F
```

### 3.3 快表（TLB）

TLB 是页表的缓存，存储最近使用的页表项：

```
CPU → 查TLB → 命中 → 直接得到物理地址
         ↓ 未命中
      查页表 → 得到物理地址 → 更新TLB
```

**多级页表**：减少页表占用的连续空间

```
一级页号 → 二级页表 → 物理帧号
例: 32位系统，页大小4KB
    一级页号(10位) → 二级页号(10位) → 页内偏移(12位)
```

### 3.4 页面置换算法

| 算法 | 原理 | 特点 |
|------|------|------|
| OPT | 替换未来最久不会被使用的页面 | 理论最优，无法实现 |
| FIFO | 替换最早进入内存的页面 | 可能出现 Belady 异常 |
| LRU | 替换最近最久未使用的页面 | 性能好，实现开销大 |
| Clock | LRU 的近似，使用引用位 | 开销适中 |
| LFU | 替换访问次数最少的页面 | 需要维护计数器 |

**Belady 异常**：增加物理页帧数反而增加缺页次数（仅 FIFO 出现）。

### 3.5 抖动（Thrashing）

当分配给进程的页帧数不足时，频繁发生缺页中断，CPU 利用率急剧下降。

**工作集模型**：
```
工作集 W(t, Δ) = { 在时刻 t 前 Δ 时间内访问过的页面集合 }

例: 内存访问序列: 1,2,5,7,8,5,1,3,7,8,7,5
    t=6, Δ=5: W(6,5) = {5,7,8,1}
    分配给进程的页帧数 ≥ 工作集大小 → 避免抖动
```

## 五、I/O 管理

### 5.1 I/O 控制方式

| 方式 | 数据传输单位 | 特点 |
|------|------------|------|
| 程序查询 | 字 | CPU 不断查询，利用率低 |
| 中断方式 | 字 | 每次传输产生一次中断 |
| DMA（直接内存访问） | 块 | 不经过CPU，效率高 |
| 通道 | 一组块 | 专用处理器，效率最高 |

### 5.2 I/O 软件层次

```
用户进程
    ↓
设备独立性软件（统一接口、缓冲管理）
    ↓
设备驱动程序（具体设备操作）
    ↓
中断处理程序（处理I/O中断）
    ↓
硬件设备
```

### 5.3 缓冲技术

**单缓冲**：操作系统在内存中分配一个缓冲区。
- 设备和CPU交替使用缓冲区

**双缓冲**：两个缓冲区。
- 设备写缓冲区A，CPU读缓冲区B，可并行

**循环缓冲**：多个缓冲区组成环形。
- 适合生产者-消费者模型

**缓冲池**：系统统一管理的缓冲区集合。
- 多个进程共享，按类型使用

## 七、操作系统经典面试题

### Q1: 进程和线程的区别？

**答**：
1. 进程是资源分配的基本单位，线程是CPU调度的基本单位
2. 进程有独立的地址空间，线程共享所属进程的地址空间
3. 进程间通信需要IPC机制，线程可直接读写共享变量
4. 创建线程的开销比创建进程小得多
5. 一个进程中的线程崩溃会导致整个进程终止

### Q2: 什么是虚拟内存？

**答**：
虚拟内存是一种内存管理技术，它为每个进程提供独立的、连续的逻辑地址空间，而物理内存可以是不连续的，甚至逻辑地址空间可以大于物理内存。通过页表将逻辑地址映射到物理地址，实现了内存的抽象和隔离。

### Q3: 什么是死锁？如何避免？

**答**：
死锁是两个或多个进程互相持有对方需要的资源，同时等待获取对方持有的资源，导致所有进程都无法继续执行。

避免死锁的方法：
1. **破坏互斥条件**：使用可共享的资源
2. **破坏请求与保持**：一次性申请所有资源
3. **破坏不可剥夺**：允许强制回收资源
4. **破坏循环等待**：资源有序分配

### Q4: LRU 缓存如何实现？

**答**：
使用 HashMap + 双向链表：
- HashMap：O(1) 查找
- 双向链表：O(1) 插入和删除

```python
class LRUCache:
    def __init__(self, capacity):
        self.capacity = capacity
        self.cache = {}       # key → Node
        self.head = Node()    # 虚拟头节点（最近使用）
        self.tail = Node()    # 虚拟尾节点（最久未使用）
        self.head.next = self.tail
        self.tail.prev = self.head

    def get(self, key):
        if key in self.cache:
            node = self.cache[key]
            self._move_to_head(node)
            return node.value
        return -1

    def put(self, key, value):
        if key in self.cache:
            node = self.cache[key]
            node.value = value
            self._move_to_head(node)
        else:
            node = Node(key, value)
            self.cache[key] = node
            self._add_to_head(node)
            if len(self.cache) > self.capacity:
                removed = self._remove_tail()
                del self.cache[removed.key]
```
