# 微服务架构详解

## 一、微服务概述

### 1.1 什么是微服务

微服务架构是一种将单一应用程序开发为一组小型服务的方法。每个服务运行在自己的进程中，服务间通过轻量级通信机制（通常是 HTTP API）进行交互。

### 1.2 单体架构 vs 微服务架构

```
单体架构:
┌─────────────────────────────────────┐
│           单一应用程序                │
│  ┌─────────┬─────────┬───────────┐ │
│  │ 用户模块 │ 订单模块 │ 支付模块  │ │
│  ├─────────┼─────────┼───────────┤ │
│  │   共享数据库 / 共享内存          │ │
│  └─────────────────────────────────┘ │
└─────────────────────────────────────┘

微服务架构:
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户服务  │  │ 订单服务  │  │ 支付服务  │
│ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │
│ │ DB 1 │ │  │ │ DB 2 │ │  │ │ DB 3 │ │
│ └──────┘ │  │ └──────┘ │  │ └──────┘ │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │
     └─────────────┴─────────────┘
              API Gateway
```

### 1.3 微服务的优缺点

| 优点 | 缺点 |
|------|------|
| 独立部署 | 分布式系统复杂性 |
| 技术异构性 | 数据一致性挑战 |
| 故障隔离 | 运维成本高 |
| 可按需扩展 | 网络延迟 |
| 团队自治 | 调试困难 |

## 二、核心组件

### 2.1 API 网关

API 网关是所有客户端请求的统一入口。

**主要功能**：
- 请求路由
- 负载均衡
- 认证鉴权
- 限流熔断
- 协议转换
- 请求聚合
- 日志记录

**主流方案对比**：

| 网关 | 特点 | 适用场景 |
|------|------|---------|
| Spring Cloud Gateway | 响应式，基于 WebFlux | Spring Cloud 生态 |
| Kong | 基于 Nginx + Lua，高性能 | 高并发场景 |
| Envoy | Cloud Native，L4/L7 | Kubernetes + Istio |
| Traefik | 自动发现，云原生 | Docker/K8s 环境 |
| APISIX | 基于 Nginx + etcd，国内社区活跃 | 国内项目 |

**Spring Cloud Gateway 配置示例**：
```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=1
            - name: CircuitBreaker
              args:
                name: user-service
                fallbackUri: forward:/fallback/user

        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1
            - AddRequestHeader=X-Source, gateway

      default-filters:
        - DedupeResponseHeader=Access-Control-Allow-Origin

      globalcors:
        cors-configurations:
          '[/**]':
            allowedOrigins: "*"
            allowedMethods: "*"
            allowedHeaders: "*"
```

### 2.2 服务注册与发现

**工作原理**：
```
服务提供者                    注册中心                    服务消费者
    │                           │                           │
    │──── 注册（IP+端口） ──────→│                           │
    │                           │                           │
    │──── 心跳续约 ────────────→│                           │
    │                           │                           │
    │                           │←── 查询服务列表 ──────────│
    │                           │──── 返回服务列表 ────────→│
    │                           │                           │
    │                           │                    负载均衡选择实例
    │                           │                           │
    │←────────────────── 直接调用 ─────────────────────────│
```

**主流注册中心对比**：

| 注册中心 | CAP 模型 | 一致性协议 | 特点 |
|---------|---------|-----------|------|
| Eureka | AP | 无（Peer-to-Peer） | Netflix 开源，最终一致 |
| Nacos | AP/CP 可切换 | Raft/Distro | 阿里开源，集成配置中心 |
| Consul | CP | Raft | HashiCorp，支持 KV |
| ZooKeeper | CP | ZAB | 强一致，维护成本高 |

### 2.3 服务间通信

#### 同步通信

**REST（HTTP/JSON）**：
```
客户端 → HTTP 请求 → 服务端
{
    "method": "GET",
    "path": "/api/users/123",
    "headers": {
        "Content-Type": "application/json"
    },
    "body": {}
}
```

**gRPC（HTTP/2 + Protobuf）**：
```protobuf
syntax = "proto3";

service UserService {
    rpc GetUser (GetUserRequest) returns (UserResponse);
    rpc CreateUser (CreateUserRequest) returns (UserResponse);
    rpc ListUsers (ListUsersRequest) returns (stream UserResponse);
}

message GetUserRequest {
    int64 id = 1;
}

message UserResponse {
    int64 id = 1;
    string name = 2;
    string email = 3;
}
```

**REST vs gRPC**：

| 特性 | REST | gRPC |
|------|------|------|
| 协议 | HTTP/1.1 | HTTP/2 |
| 数据格式 | JSON（文本） | Protobuf（二进制） |
| 性能 | 一般 | 高（3-10倍） |
| 流式支持 | 有限 | 原生支持 |
| 浏览器支持 | 直接支持 | 需要 gRPC-Web |
| 学习曲线 | 低 | 中 |
| 调试 | 方便 | 较复杂 |

#### 异步通信（消息队列）

```
生产者 → 消息队列（Kafka/RabbitMQ/RocketMQ）→ 消费者
```

**消息队列选型**：

| 特性 | Kafka | RabbitMQ | RocketMQ |
|------|-------|----------|----------|
| 吞吐量 | 极高（百万级/s） | 中等（万级/s） | 高（十万级/s） |
| 延迟 | ms级 | μs级 | ms级 |
| 消息可靠性 | 高 | 高 | 高 |
| 顺序消息 | 支持 | 不保证 | 支持 |
| 事务消息 | 支持 | 不支持 | 支持 |
| 延迟消息 | 不支持 | 插件支持 | 支持 |
| 适用场景 | 日志、大数据 | 业务消息 | 电商、金融 |

### 2.4 熔断器

**状态机模型**：
```
关闭状态（Closed）
    │
    │ 失败率超过阈值
    ▼
打开状态（Open）
    │
    │ 超时后
    ▼
半开状态（Half-Open）
    │
    ├── 探测成功 → 关闭状态
    │
    └── 探测失败 → 打开状态
```

**Resilience4j 配置**：
```java
// 熔断器配置
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)           // 失败率阈值 50%
    .waitDurationInOpenState(Duration.ofSeconds(10)) // 打开状态持续时间
    .slidingWindowSize(10)              // 滑动窗口大小
    .minimumNumberOfCalls(5)            // 最小调用次数
    .permittedNumberOfCallsInHalfOpenState(3) // 半开状态允许的调用次数
    .build();

// 重试配置
RetryConfig retryConfig = RetryConfig.custom()
    .maxAttempts(3)
    .waitDuration(Duration.ofMillis(500))
    .retryExceptions(RuntimeException.class)
    .build();

// 限流配置
RateLimiterConfig rateLimiterConfig = RateLimiterConfig.custom()
    .limitForPeriod(10)               // 每周期允许的调用次数
    .limitRefreshPeriod(Duration.ofSeconds(1))
    .timeoutDuration(Duration.ofMillis(500))
    .build();

// 组合使用
Supplier<String> decoratedSupplier = Decorators
    .ofSupplier(this::remoteCall)
    .withCircuitBreaker(circuitBreaker)
    .withRetry(retry)
    .withRateLimiter(rateLimiter)
    .decorate();

Try<String> result = Try.ofSupplier(decoratedSupplier)
    .recover(CallNotPermittedException.class, e -> "降级响应")
    .recover(TimeoutException.class, e -> "超时降级");
```

### 2.5 限流算法

```go
// 令牌桶算法
type TokenBucket struct {
    rate     int       // 每秒产生的令牌数
    capacity int       // 桶容量
    tokens   int       // 当前令牌数
    lastTime time.Time // 上次填充时间
}

func (tb *TokenBucket) Allow() bool {
    now := time.Now()
    // 计算并添加新令牌
    elapsed := now.Sub(tb.lastTime).Seconds()
    newTokens := int(elapsed * float64(tb.rate))
    tb.tokens = min(tb.capacity, tb.tokens+newTokens)
    tb.lastTime = now

    if tb.tokens > 0 {
        tb.tokens--
        return true
    }
    return false
}

// 滑动窗口计数器
type SlidingWindow struct {
    windowSize time.Duration
    maxCount   int
    windows    map[int64]int // 时间戳 → 计数
}

func (sw *SlidingWindow) Allow() bool {
    now := time.Now().Unix()
    windowKey := now / int64(sw.windowSize.Seconds())

    var count int
    for k, v := range sw.windows {
        if k >= windowKey-1 {
            count += v
        }
    }

    if count < sw.maxCount {
        sw.windows[windowKey]++
        return true
    }
    return false
}
```

## 三、数据管理

### 3.1 数据库拆分

**按业务拆分（垂直拆分）**：
```
用户服务 → MySQL (user_db)
订单服务 → MySQL (order_db)
商品服务 → MySQL (product_db)
支付服务 → MySQL (payment_db)
```

**按数据量拆分（水平拆分）**：
```
订单表
├── order_db_0 (订单ID % 4 == 0)
├── order_db_1 (订单ID % 4 == 1)
├── order_db_2 (订单ID % 4 == 2)
└── order_db_3 (订单ID % 4 == 3)
```

### 3.2 分布式事务

#### CAP 定理
- **C（Consistency）**：一致性
- **A（Availability）**：可用性
- **P（Partition tolerance）**：分区容错性

分布式系统只能同时满足其中两个（通常选择 AP + 最终一致性）。

#### 最终一致性方案

**本地消息表**：
```
1. 业务操作 + 写消息表（同一事务）
2. 定时任务扫描消息表，发送到消息队列
3. 消费者处理消息，更新业务数据
4. 处理成功后标记消息为已完成
```

**Saga 模式**：
```
正向操作:
T1 (创建订单) → T2 (扣减库存) → T3 (扣减余额) → T4 (创建支付)

补偿操作（失败时回滚）:
C1 (取消订单) ← C2 (恢复库存) ← C3 (恢复余额) ← C4 (取消支付)
```

**Seata 分布式事务**：
```yaml
# Seata 配置
seata:
  tx-service-group: my_tx_group
  service:
    vgroup-mapping:
      my_tx_group: default
  registry:
    type: nacos
    nacos:
      server-addr: 127.0.0.1:8848
```

```java
@GlobalTransactional(timeoutMills = 60000, name = "create-order")
public void createOrder(OrderDTO orderDTO) {
    // 1. 创建订单
    orderService.create(orderDTO);

    // 2. 扣减库存（远程调用）
    storageService.deduct(orderDTO.getProductId(), orderDTO.getCount());

    // 3. 扣减余额（远程调用）
    accountService.debit(orderDTO.getUserId(), orderDTO.getMoney());
}
```

## 七、微服务拆分原则

### 7.1 拆分策略

| 策略 | 说明 |
|------|------|
| 按业务能力拆分 | 按业务领域划分（用户、订单、商品） |

### 7.2 DDD（领域驱动设计）

```
限界上下文（Bounded Context）
├── 用户上下文
│   ├── 实体: User

└── 支付上下文

    ├── 值对象: PaymentMethod
    └── 聚合根: PaymentAggregate
```

## 九、微服务面试高频问题

### Q1: 如何保证分布式事务一致性？

**答**：
- 强一致性：2PC、3PC（性能差，不推荐）
- 最终一致性：本地消息表、Saga、TCC、最大努力通知
- 推荐使用 Seata 的 AT 模式或 RocketMQ 事务消息

### Q2: 服务雪崩如何解决？

**答**：
1. 熔断：快速失败，避免级联调用
2. 限流：控制请求速率
3. 隔离：线程池/信号量隔离
4. 降级：返回兜底数据
5. 超时控制：设置合理的超时时间

### Q3: 微服务如何测试？

**答**：
- 单元测试：服务内部逻辑
- 集成测试：服务间调用
- 契约测试：Pact/Spring Cloud Contract
- 端到端测试：完整链路
- 压测：JMeter/Gatling
- 线上演练：混沌工程（Chaos Monkey）
