# 软件工程知识问答平台设计与实现

## 摘要

随着软件工程学科的快速发展和教育信息化的深入推进，传统教学模式在知识体系可视化、个性化学习支持和智能答疑等方面存在不足，难以满足学生自主学习和教师精准教学的需求。如何借助人工智能技术构建智能化的教学辅助平台，提升软件工程课程的教学质量和学习效率，已成为当前教育技术领域的重要研究课题。

针对上述问题，本文设计并实现了一个面向软件工程课程的"SE智图问答"平台。该平台采用 Vue 3 + TypeScript + Element Plus 构建前端，基于 Go Gin 框架开发后端，数据存储采用 MySQL + Neo4j + MinIO 的混合方案。平台集成 Ollama 本地部署的大语言模型（Qwen3:8b）和文本嵌入模型（Nomic-embed-text），实现了基于检索增强生成（RAG）的智能问答功能。平台包含学生和教师两种角色，学生端主要提供知识问答、知识图谱浏览、题库练习和学习分析等功能，教师端主要提供知识图谱自动构建、题库管理和资料审核等功能。

经过黑盒测试验证，本平台完成了注册登录、智能问答、知识图谱可视化与构建、题库练习、学习分析、资料管理等核心功能的开发与测试，各项功能运行正常，系统整体稳定可靠，能够满足软件工程课程教学中知识管理、智能答疑和辅助学习的基本需求。

**关键词：** 知识图谱；智能问答；大语言模型；检索增强生成；软件工程教育

# 1 绪论

## 1.1 研究背景及意义

随着软件工程学科的快速发展和教育信息化的深入推进，传统的教学模式面临着诸多挑战。在软件工程课程教学中，知识体系庞杂、概念抽象、实践性强等特点使得学生难以系统掌握课程内容。传统的"填鸭式"教学方式已无法满足现代教育的需求，亟需一种能够支持个性化学习、知识可视化和智能问答的辅助教学平台。

知识图谱（Knowledge Graph）作为一种新兴的知识组织和表示技术，能够将碎片化的知识结构化、网络化，为知识管理和智能问答提供了新的思路。将知识图谱技术应用于软件工程课程教学，可以构建课程知识体系的可视化表示，帮助学生建立知识间的关联，提升学习效率。

同时，随着人工智能技术的成熟，智能问答系统在教育领域的应用日益广泛。基于知识图谱的问答系统能够理解用户问题的语义，精准定位相关知识，提供准确、可解释的答案，对于辅助学生自主学习具有重要价值。

此外，题库练习和学习分析也是教育平台不可或缺的功能模块。通过对学生答题行为的记录和分析，系统能够识别学生的薄弱知识点，为教师提供教学决策支持，实现精准教学。

本项目旨在设计并实现一个面向软件工程课程的"SE智图问答"平台，集成知识图谱可视化、智能问答、题库练习、学习分析等功能，为学生提供一个智能化、个性化的学习辅助工具，为教师提供教学管理和学情分析工具。该平台的研究与实现对于推动软件工程课程教学改革、提升教学质量具有重要的理论意义和实践价值。

## 1.2 国内外研究现状

### 1.2.1 国内研究现状与分析

国内在教育知识图谱和智能问答系统方面的研究起步较晚，但发展迅速。近年来，众多学者和研究机构在该领域开展了大量研究工作，取得了一系列成果。

（1）教育知识图谱研究

在教育知识图谱构建方面，国内研究主要集中在知识抽取、知识表示和知识推理等关键技术上。研究者们提出了多种基于深度学习的知识抽取方法，能够从非结构化文本中自动提取实体和关系。例如，基于BERT[4]的命名实体识别和关系抽取模型在教育领域知识图谱构建中表现出较高的准确率。严蔚敏等[1]在数据结构教材中系统阐述了图论等基础知识，为知识图谱中图结构的表示和算法提供了理论基础。Hogan等[5]在知识图谱的综合综述中系统总结了知识表示、存储和查询的核心技术，为教育知识图谱的构建提供了方法论指导。

（2）智能问答系统研究

在智能问答方面，国内学者提出了多种基于知识图谱的问答方法。这些方法主要包括基于模板匹配的方法、基于语义解析的方法和基于深度学习的方法。基于深度学习的方法利用注意力机制、图神经网络等技术，能够更好地理解问题语义并从知识图谱中检索答案。Kipf等[6]提出的图卷积网络（GCN）为知识图谱上的信息传播和推理提供了有效的图神经网络框架，被广泛应用于知识图谱问答任务中。Pan等[7]对大语言模型与知识图谱的融合进行了系统性综述，指出知识图谱能够为大模型提供结构化的外部知识，增强问答系统的准确性和可解释性。

（3）教育平台实践

在应用层面，国内已有多款基于知识图谱的教育产品问世。这些产品主要应用于自适应学习、智能推荐、学情分析等场景。例如，某些在线教育平台利用知识图谱技术为学生推荐个性化学习路径，根据学生的知识掌握情况动态调整学习内容，取得了良好的教学效果。

### 1.2.2 国外研究现状与分析

国外在教育知识图谱和智能问答系统方面的研究起步较早，技术相对成熟。欧美等发达国家的高校和研究机构在该领域积累了丰富的研究成果。

（1）知识图谱技术研究

在知识图谱技术方面，国外研究者提出了多种经典的知识表示方法和图谱构建框架。Google于2012年推出的知识图谱项目是该领域的里程碑事件，其"Things, not strings"的理念深刻影响了后续的知识表示研究。DBpedia、Wikidata、YAGO等大型开放知识图谱的构建，为教育领域提供了宝贵的知识资源和实践经验。Bordes等[8]提出的TransE模型开创了知识图谱嵌入表示的先河，通过将实体和关系映射到低维向量空间，实现了知识的高效表示和推理。Hogan等[5]在其综合综述中系统梳理了知识图谱从构建、存储到查询和推理的完整技术栈，为后续研究提供了重要的参考框架。

（2）智能教育系统

在智能教育系统方面，国外已有多款成熟的产品和平台。Carnegie Learning的智能辅导系统利用知识追踪技术，实时评估学生的知识状态并调整教学策略。Piech等[9]提出的深度知识追踪（Deep Knowledge Tracing）模型，利用循环神经网络对学生的学习过程进行建模，能够更准确地预测学生对知识点的掌握程度，开创了智能教育领域知识追踪的新范式。Khan Academy的个性化学习平台通过知识图谱组织课程内容，为学生提供自适应的学习路径推荐。这些系统集成了知识图谱、自然语言处理、机器学习等前沿技术，为学生提供智能化的学习支持。

（3）知识图谱在教育中的应用研究

在学术研究方面，国外学者在教育知识图谱的构建方法、知识推理算法、问答系统架构等方面开展了深入研究。Speer等[10]构建的ConceptNet是一个大规模的常识知识图谱，将词汇语义网络与知识图谱相结合，为教育领域的语义理解和知识关联提供了丰富的常识知识基础。研究表明，基于知识图谱的教学系统能够显著提升学生的学习效果和学习兴趣，为个性化教育提供了有力的技术支撑。Devlin等[4]提出的BERT预训练语言模型在多项自然语言处理任务中取得了突破性成绩，其在教育知识图谱问答中的应用研究也成为近年来的热点方向。

# 2 相关技术

## 2.1 前端技术

本平台前端采用 Vue 3 框架构建单页面应用（SPA），主要技术选型如下：

（1）Vue 3

Vue 3 是一款渐进式 JavaScript 框架，采用 Composition API 和 `<script setup>` 语法，提供了更好的代码组织和类型推导能力。相比 Vue 2，Vue 3 在性能上有了显著提升，包括更快的虚拟 DOM diff 算法、Tree-shaking 支持和更好的 TypeScript 集成。

（2）TypeScript

TypeScript 是 JavaScript 的超集，增加了静态类型检查能力。在大型前端项目中，TypeScript 能够有效减少运行时错误，提高代码的可维护性和可读性。本项目全面采用 TypeScript 进行开发，确保代码质量。

（3）Element Plus

Element Plus 是基于 Vue 3 的桌面端 UI 组件库，提供了丰富的表单、表格、对话框、导航等组件。本项目使用 Element Plus 构建用户界面，保证了 UI 的一致性和开发效率。

（4）Vue Router

Vue Router 是 Vue.js 的官方路由管理器，支持嵌套路由、路由守卫、懒加载等特性。本项目使用 Vue Router 实现页面路由管理，通过路由守卫实现登录鉴权。

（5）Pinia

Pinia 是 Vue 的新一代状态管理库，替代了 Vuex。它提供了更简洁的 API、完整的 TypeScript 支持和 DevTools 集成。本项目使用 Pinia 管理用户登录状态、全局配置等共享状态。

（6）Axios

Axios 是一个基于 Promise 的 HTTP 客户端，支持请求/响应拦截、自动转换 JSON 数据等功能。本项目使用 Axios 封装统一的 API 请求模块，实现与后端的数据交互。

（7）D3.js

D3（Data-Driven Documents）是一个用于数据可视化的 JavaScript 库，提供了丰富的数据驱动文档操作方法。本项目使用 D3.js 实现知识图谱的力导向布局渲染，展示知识点之间的关系网络。

（8）ECharts

ECharts 是百度开源的可视化库，提供了丰富的图表类型。本项目使用 ECharts 绘制学习分析统计图表，如柱状图、饼图、雷达图等。

## 2.2 后端技术

本平台后端采用 Go 语言开发，主要技术选型如下：

（1）Go 语言

Go 是 Google 开发的静态强类型编译型语言，具有并发性能优异、编译速度快、部署简便等特点。Go 的 goroutine 和 channel 机制为高并发场景提供了原生支持，适合构建高性能的 Web 服务。

（2）Gin 框架

Gin 是一个基于 Go 的高性能 Web 框架，采用 httprouter 进行路由匹配，性能优异。Gin 提供了中间件支持、参数绑定、JSON 验证、分组路由等特性，大大简化了 Web API 的开发。本项目使用 Gin 框架构建 RESTful API。

（3）GORM

GORM 是 Go 语言的 ORM（对象关系映射）库，支持 MySQL、PostgreSQL、SQLite 等多种数据库。GORM 提供了完整的 CRUD 操作、关联查询、事务管理、自动迁移等功能，简化了数据库操作。本项目使用 GORM 作为数据库访问层。

（4）JWT（JSON Web Token）

JWT 是一种基于 JSON 的开放标准（RFC 7519），用于在网络应用环境间安全地传递声明信息。本项目使用 JWT 实现用户身份认证，通过 golang-jwt 库生成和验证 Token。

（5）bcrypt 密码加密

本项目使用 Go 标准库中的 bcrypt 算法对用户密码进行哈希处理，确保用户密码的安全存储。bcrypt 算法具有自适应计算复杂度的特点，能够有效抵御暴力破解攻击。

（6）Ollama 本地大语言模型推理引擎

Ollama 是一款轻量级的本地大语言模型（LLM）推理引擎，支持在本地环境运行开源大模型，无需依赖云端 API 服务。本平台通过 Ollama 部署了 Qwen3:8b 作为文本生成模型，用于智能问答、知识抽取等 AI 功能。Ollama 提供了兼容 OpenAI 的 RESTful API 接口，后端通过自定义的 OllamaClient 封装了文本生成（Generate）、流式生成（GenerateStream）和文本嵌入（Embed）等核心 API 调用。流式生成基于 HTTP Streaming 和 Server-Sent Events（SSE）机制，通过 Go 的 goroutine 和 channel 实现异步 token 流传输，支持实时逐字输出回答内容。

（7）RAG（检索增强生成）

RAG（Retrieval-Augmented Generation）是一种将信息检索与大语言模型生成相结合的技术范式。本平台实现了完整的 RAG 管道：首先通过向量检索从知识库中找到与用户问题最相关的文档片段，然后将检索到的上下文与用户问题一起输入大语言模型，生成基于知识库的准确回答。RAG 技术有效解决了大语言模型"幻觉"问题，确保回答内容有据可依。后端的 AnswerService 封装了 RAG 问答的核心逻辑，支持多轮对话历史上下文的维护。

（8）文本嵌入与向量检索

文本嵌入（Text Embedding）是将自然语言文本映射到高维向量空间的技术，使得语义相似的文本在向量空间中距离更近。本平台使用 Ollama 部署的 Nomic-embed-text 嵌入模型（768 维），将课程资料文档分块后转换为向量表示并存储在内存向量索引中。查询时，使用余弦相似度（Cosine Similarity）计算用户问题向量与文档向量的相似度，返回最相关的 Top-K 个文档片段作为 RAG 检索结果。向量索引支持持久化存储和增量更新。

（9）LLM 知识抽取

本平台利用大语言模型的自然语言理解能力，实现从课程资料文档中自动抽取知识点和知识点关系的功能。抽取服务（ExtractionService）通过精心设计的 Prompt 模板，引导 LLM 从非结构化文本中识别实体（知识点）和关系（知识点间的语义联系），并将抽取结果结构化为 JSON 格式。当 LLM 服务不可用时，系统自动降级为基于正则表达式和软件工程本体关键词的规则抽取方案，保证系统的可用性。

## 2.3 数据库技术

本平台采用多种数据库技术相结合的存储方案，根据不同数据的特点选择最适合的存储引擎。

（1）MySQL

MySQL 是全球最流行的开源关系型数据库管理系统，具有稳定可靠、性能优异、社区活跃等优点。MySQL 支持事务处理、外键约束、索引优化等特性，能够满足本平台结构化业务数据的存储需求。本平台使用 MySQL 存储用户信息、题目、答题记录、问答会话、文档元数据等关系型数据。数据库采用 utf8mb4 字符集和 utf8mb4_unicode_ci 排序规则，支持完整的 Unicode 字符，确保多语言内容的正确存储和检索。

（2）Neo4j 图数据库

Neo4j 是一款基于属性图模型的原生图数据库，采用免索引邻接（Index-Free Adjacency）存储机制，在图遍历和路径查询方面具有显著的性能优势。本平台使用 Neo4j 存储知识图谱中的节点（知识点）和边（知识点关系），支持高效的关系路径查询和图遍历操作。相比关系型数据库中通过多表 JOIN 模拟图查询，Neo4j 能够以常数时间复杂度完成节点间的关系遍历，特别适合知识图谱中复杂关系网络的存储和查询。后端通过 neo4j-go-driver 驱动与 Neo4j 数据库进行通信，支持 Cypher 查询语言进行图数据的增删改查操作。

（3）MinIO 对象存储

MinIO 是一款高性能、S3 兼容的分布式对象存储系统，适用于非结构化数据的存储管理。本平台使用 MinIO 存储用户上传的课程资料文件（如 PDF、DOCX、TXT 等），替代传统的本地文件系统存储方案。MinIO 提供了文件上传、下载、删除、预签名 URL 生成等完整的对象操作 API，支持大文件分片上传和元数据管理。后端通过 minio-go SDK 与 MinIO 服务进行交互，通过 Bucket（存储桶）对文件进行逻辑隔离和管理。

# 3 需求分析

## 3.1 用例分析

本平台涉及两类用户角色：学生（User）和教师（Teacher），各角色的功能用例如下：

### 3.1.1 学生用例

（1）注册/登录：学生通过填写用户名、邮箱、密码进行注册，注册成功后可使用账号密码登录系统。

（2）知识问答：学生输入软件工程相关问题，系统基于关键词匹配和知识图谱检索返回答案、置信度评分及相关知识点引用。

（3）知识图谱浏览：学生以可视化方式浏览课程知识图谱，查看知识点及其关系，支持节点展开、缩放等交互操作。

（4）资料管理：学生可上传课程资料（支持 PDF、DOCX、TXT 等格式），查看资料列表和处理状态。

（5）题库练习：学生从题库中选择题目进行练习，提交答案后获得评分和详细解析。

（6）学习分析：学生查看个人学习统计数据，包括问答次数、答题正确率、薄弱知识点分析等。

（7）个人中心：学生可查看和修改个人信息，包括昵称、头像等。

### 3.1.2 教师用例

（1）教师登录：教师使用专属账号登录教师端管理系统。

（2）知识图谱构建：教师选择课程资料，触发知识图谱自动构建，系统从资料中抽取知识点和关系并写入知识图谱数据库。

（3）题库管理：教师可查看、添加、编辑和删除题库中的题目。

（4）资料审核：教师可审核学生上传的资料，通过审核的资料可用于知识图谱构建。

（5）学情分析：教师查看学生的学习数据统计，了解整体学情。

## 3.2 交互分析

### 3.2.1 学生端交互流程

（1）登录/注册流程：学生访问平台首页，未登录状态自动跳转至登录页面。学生输入用户名和密码进行登录，登录成功后跳转至首页。若未注册，可点击注册链接跳转至注册页面。

（2）知识问答交互：学生在问答中心页面输入问题，系统实时检索知识库并返回答案。答案包含回答内容、置信度评分和相关知识点链接。

（3）知识图谱交互：学生进入知识图谱页面，系统加载并渲染图谱数据。学生可通过鼠标拖拽、滚轮缩放等操作浏览图谱，点击节点查看知识点详情。

（4）答题交互：学生进入题库练习页面，选择题目类型和难度，系统随机或按序展示题目。学生选择答案后提交，系统即时返回评分和解析。

### 3.2.2 教师端交互流程

（1）教师登录流程：教师访问教师端登录页面，输入教师账号密码登录。

（2）图谱构建交互：教师在管理页面选择用于构建图谱的资料文档，点击"构建图谱"按钮。系统异步执行构建任务，完成后展示构建结果（新增知识点数、新增关系数等）。

## 3.3 分析类

根据需求分析，本平台涉及以下核心分析类：

（1）User（用户类）：表示学生用户实体，包含用户名、密码、邮箱、昵称、头像、状态等属性，以及注册、登录、信息修改等操作。

（2）Teacher（教师类）：表示教师用户实体，包含用户名、密码、邮箱、昵称、头像、状态等属性，以及登录、信息修改等操作。

（3）KnowledgePoint（知识点类）：表示知识图谱中的一个节点，包含名称、描述、分类、关联文档等属性。

（4）KnowledgeRelation（知识点关系类）：表示知识图谱中两个知识点之间的关系，包含源节点、目标节点、关系类型、描述等属性。

（5）KnowledgeBuild（图谱构建记录类）：记录每次知识图谱构建的统计信息，包含文档列表、构建状态、新增知识点数、新增关系数等。

（6）Question（题目类）：表示试题实体，包含标题、类型、难度、关联知识点、选项、正确答案、解析等属性。

（7）Quiz（答题记录类）：记录用户的答题结果，包含题目、用户、用户答案、是否正确等属性。

（8）AskSession（问答会话类）：表示一次问答会话，包含用户、会话标题等属性。

（9）AskMessage（问答消息类）：表示问答中的一条消息，包含会话、角色、内容、置信度等属性。

（10）Document（文档类）：表示上传的课程资料，包含标题、文件名、文件路径、文件大小、解析内容、处理状态等属性。

# 4 总体设计

## 4.1 体系结构设计

本平台采用前后端分离的 B/S（Browser/Server）架构，分为三个层次：

（1）表现层（Presentation Layer）：基于 Vue 3 的单页面应用，负责用户界面的渲染和交互。前端通过 Axios 以 HTTP 请求方式调用后端 API，获取数据并渲染到页面上。

（2）业务逻辑层（Business Logic Layer）：基于 Go Gin 框架构建的 RESTful API 服务，负责处理业务逻辑。后端按 MVC 模式组织代码，分为 Controller（控制器）、Service（服务层）、Repository（数据访问层）三个子层。

- Controller 层：负责接收 HTTP 请求，进行参数绑定和验证，调用 Service 层处理业务逻辑，返回统一格式的 JSON 响应。
- Service 层：封装核心业务逻辑，如知识图谱构建、问答检索、答题评分等。
- Repository 层：封装数据库操作，使用 GORM 进行 CRUD 操作和复杂查询。

（3）数据持久层（Data Persistence Layer）：采用 MySQL 关系型数据库，存储用户数据、知识点、题目、答题记录、文档等业务数据。

# 5 详细设计与实现

## 5.1 学生端功能

### 5.1.1 注册

（1）界面设计与实现效果

注册页面采用简洁的居中卡片式布局，包含用户名、邮箱、密码和确认密码四个输入框，以及"注册"按钮和"已有账号？去登录"的跳转链接。页面整体风格以白色和灰色为主色调，符合极简设计理念。用户名和邮箱输入框支持实时格式校验，密码输入框支持显示/隐藏密码切换。

（2）注册功能实现过程

学生在注册页面填写用户名、邮箱、密码和确认密码后，前端首先进行表单验证，检查各字段是否符合格式要求（用户名长度、邮箱格式、密码强度、两次密码一致性）。验证通过后，前端通过 Axios 向后端发送 POST 请求至 `/api/v1/auth/register` 接口。

后端 Controller 层接收请求后，将请求体绑定到注册 DTO 结构体，进行参数验证。验证通过后，调用 Service 层的注册逻辑：首先检查用户名和邮箱是否已被注册，若未被占用，则使用 bcrypt 算法对密码进行哈希处理，创建 User 实体并调用 Repository 层写入数据库。注册成功后返回统一格式的成功响应，前端收到响应后自动跳转至登录页面。

核心代码逻辑如下：

```go
// Service 层注册逻辑
func (s *AuthService) Register(req *dto.RegisterRequest) error {
    // 1. 检查用户名是否已存在
    if existing, _ := s.userRepo.FindByUsername(req.Username); existing != nil {
        return errors.New("用户名已被注册")
    }
    // 2. 检查邮箱是否已存在
    if existing, _ := s.userRepo.FindByEmail(req.Email); existing != nil {
        return errors.New("邮箱已被注册")
    }
    // 3. 密码哈希
    hashedPassword, _ := bcrypt.GenerateFromPassword([]byte(req.Password), bcrypt.DefaultCost)
    // 4. 创建用户并写入数据库
    user := &entity.User{
        Username: req.Username,
        Password: string(hashedPassword),
        Email:    req.Email,
        Nickname: req.Nickname,
    }
    return s.userRepo.Create(user)
}
```

### 5.1.2 登录

（1）界面设计与实现效果

登录页面采用与注册页面一致的居中卡片式布局，包含用户名和密码两个输入框，以及"登录"按钮和"没有账号？去注册"的跳转链接。页面设计简洁，输入框支持回车提交。

（2）登录功能实现过程

学生在登录页面输入用户名和密码后，前端通过 Axios 向后端发送 POST 请求至 `/api/v1/auth/login` 接口。后端 Controller 层接收请求后，将请求体绑定到登录 DTO 结构体，调用 Service 层的登录逻辑。

Service 层首先根据用户名查询用户记录，若用户不存在则返回错误。用户存在后，使用 bcrypt.CompareHashAndPassword 函数比对密码哈希。密码验证通过后，使用 golang-jwt 库生成 JWT Token，将用户 ID 和用户名编码到 Token 中，并设置过期时间。最后将 Token 和用户基本信息返回给前端。

前端收到响应后，将 Token 存储到 Pinia 状态管理和 localStorage 中，后续请求通过 Axios 请求拦截器自动携带 Token。登录成功后，前端路由守卫检测到有效 Token，允许用户访问受保护的页面。

```go
// Service 层登录逻辑
func (s *AuthService) Login(req *dto.LoginRequest) (string, *entity.User, error) {
    // 1. 根据用户名查询用户
    user, err := s.userRepo.FindByUsername(req.Username)
    if err != nil {
        return "", nil, errors.New("用户名或密码错误")
    }
    // 2. 验证密码
    if err := bcrypt.CompareHashAndPassword([]byte(user.Password), []byte(req.Password)); err != nil {
        return "", nil, errors.New("用户名或密码错误")
    }
    // 3. 生成 JWT Token
    token, err := generateToken(user.ID, user.Username)
    if err != nil {
        return "", nil, err
    }
    return token, user, nil
}
```

前端接收到数据后，使用 D3.js 的 forceSimulation 模块创建力导向图模拟。将知识点映射为 SVG 圆形节点，将关系映射为 SVG 线段，通过 D3 的数据绑定机制将图谱数据渲染到 SVG 画布上。力模拟中的力包括：

- charge（电荷力）：使节点相互排斥，防止重叠。
- link（链接力）：沿关系连线吸引相关节点。
- center（中心力）：将图谱整体居中。

系统异步执行构建任务，页面显示构建进度提示。构建完成后，对话框展示构建结果统计：新增知识点数量、新增关系数量、跳过的重复知识点数量。构建完成后自动刷新图谱可视化。

（2）知识图谱构建功能实现过程

教师选择文档并触发构建后，前端通过 Axios 向后端发送 POST 请求至 `/api/v1/graph/build` 接口，请求体包含所选文档 ID 列表。

后端 Service 层执行以下构建流程：

1. 根据文档 ID 列表查询 Document 记录，获取文档的解析内容。
2. 对文档内容进行分块处理，提取关键知识点和关系。
3. 使用确定性规则从文档中抽取知识点（实体）和知识点间的关系。
4. 对抽取的知识点进行去重检查，与 KnowledgePoint 表中已有的知识点比对，跳过重复项。
5. 将新的知识点写入 KnowledgePoint 表，新的关系写入 KnowledgeRelation 表。
6. 记录本次构建的统计信息到 KnowledgeBuild 表。
7. 返回构建结果（新增知识点数、新增关系数、跳过数）。

构建完成后，前端自动调用 GET `/api/v1/graph` 接口刷新图谱数据并重新渲染可视化。

## 5.3 公共功能

### 5.3.1 JWT 鉴权中间件

本平台使用 JWT（JSON Web Token）实现身份认证。后端定义了 `RequireAuth` 和 `RequireTeacherAuth` 两个 Gin 中间件，分别用于学生接口和教师接口的鉴权。

`RequireAuth` 中间件从请求头的 `Authorization` 字段中提取 Bearer Token，使用 golang-jwt 库进行解析和验证。Token 验证通过后，将用户 ID 和用户名写入 Gin 上下文（c.Set），供后续的 Controller 和 Service 使用。Token 无效或过期时，返回 401 未授权响应。

`RequireTeacherAuth` 中间件的逻辑与 `RequireAuth` 类似，但验证的是 Teacher 身份的 Token。

### 5.3.2 文件上传与管理

本平台支持课程资料的上传和管理。前端通过 Element Plus 的 Upload 组件实现文件选择和上传，后端使用 Gin 的 `multipart/form-data` 解析上传的文件。

文件上传流程如下：

1. 前端将文件通过 FormData 封装，通过 Axios POST 请求发送至 `/api/v1/document/upload` 接口。
2. 后端解析上传的文件，获取文件名、文件大小、文件类型等信息。
3. 使用 `os.MkdirAll` 创建上传目录（若不存在），使用 `os.Create` 创建目标文件。
4. 使用 `io.Copy` 将上传的文件内容写入目标文件。
5. 将文件元数据（标题、文件名、文件路径、文件大小等）保存到 Document 表中。
6. 返回文档信息给前端。

# 6 测试

## 6.0 测试方法介绍

本平台主要采用黑盒测试（Black-box Testing）技术对系统功能进行验证。黑盒测试是一种基于功能规格说明的测试方法，测试人员无需了解程序的内部逻辑结构和代码实现，仅从用户的角度出发，将被测系统视为一个"黑盒"，通过设计不同的输入数据来验证系统的输出结果是否符合预期[11]。

黑盒测试的核心思想是根据需求规格说明设计测试用例，检查系统的以下方面：

（1）功能正确性：验证系统是否正确实现了需求中定义的各项功能，包括正常的业务流程和异常的错误处理流程。

（2）接口规范性：验证系统的 API 接口是否按照设计规范处理请求参数，包括参数类型校验、必填字段校验、边界值校验等。

（3）数据完整性：验证系统是否正确地将数据持久化到数据库中，以及数据的一致性和完整性是否得到保障。

（4）错误处理能力：验证系统在异常输入、越权访问、资源不存在等边界条件下是否能返回合理的错误响应，而不是崩溃或暴露内部信息。

本平台选择黑盒测试的原因在于：本平台采用前后端分离的 B/S 架构，后端以 RESTful API 的形式提供服务，各接口的输入输出格式明确，非常适合通过构造 HTTP 请求来验证接口的行为。测试过程中，使用 Postman 和 curl 工具向后端 API 发送不同参数的请求，检查响应状态码和响应体内容是否符合预期。

# 7 结论与展望

## 7.1 总结

本项目围绕软件工程课程教学的实际需求，设计并实现了一个集知识图谱可视化、智能问答、题库练习和学习分析于一体的"SE智图问答"平台。平台采用前后端分离的 B/S 架构，前端基于 Vue 3 + TypeScript + Element Plus 构建，后端基于 Go Gin 框架开发，采用 MVC 分层模式组织代码。数据存储方面采用 MySQL + Neo4j + MinIO 的混合方案，分别存储业务数据、知识图谱和文件资源。

在功能实现方面，平台面向学生和教师两类角色提供了完整的功能体系。学生端包括基于 RAG 的智能问答（集成 Ollama 本地部署的 Qwen3:8b 大语言模型）、知识图谱可视化浏览、题库练习和学习分析等功能。教师端包括知识图谱自动构建（利用 LLM 从文档中抽取知识点和关系）、题库管理和资料审核等功能。系统具备完善的降级机制，当 AI 服务不可用时自动切换为关键词匹配方案，保证服务可用性。

在技术应用方面，本项目实践了本地大语言模型部署、检索增强生成（RAG）、多数据库混合存储等关键技术。通过 Ollama 引擎在本地部署 LLM，兼顾了数据隐私和调用性能；通过向量检索 + LLM 生成的 RAG 管道，有效缓解了大模型的"幻觉"问题。

在测试验证方面，采用黑盒测试技术设计了 30 余个测试用例，覆盖注册、登录、问答、答题、资料上传、图谱构建等核心功能，涵盖正常流程和异常边界条件。测试结果表明各功能模块运行正常，系统整体稳定可靠。

## 7.2 展望

尽管本平台已实现了预期的功能目标，但受限于开发周期和硬件条件，系统仍有较大的优化空间。以下从模型优化、功能完善和技术改进三个方面对未来工作进行展望。

（1）Prompt 优化与知识抽取增强：当前的知识图谱构建依赖正则表达式和本体关键词规则，抽取精度有限。未来可在现有模型基础上，通过优化 Prompt 模板设计提升 LLM 的知识抽取效果，例如设计更精细的抽取指令，引导模型直接输出结构化的知识点三元组，减少后处理步骤。同时可探索 Few-shot Learning 方式，用少量标注样例引导模型提升抽取准确率。

（2）个性化学习推荐：当前平台缺乏个性化推荐功能。未来可基于学生的答题记录和薄弱知识点分析，利用简单的协同过滤或基于规则的推荐策略，为学生推荐针对性的练习题目和学习内容，实现因材施教。

（3）RAG 检索质量提升：当前的向量检索使用内存索引和简单的余弦相似度计算。未来可优化文档分块策略（如按语义段落分块而非固定长度分块），引入混合检索（向量检索 + 关键词检索）提升检索召回率，改善问答的回答质量。

（4）多终端适配：当前平台主要面向桌面端设计。未来可采用响应式布局优化移动端体验，支持学生通过手机、平板等设备随时随地学习。

（5）Docker 容器化部署：当前平台的部署依赖手动安装 Go、MySQL、Neo4j、Ollama 等环境。未来可编写 Docker Compose 配置文件，一键启动所有服务，降低部署门槛，方便在不同机器上快速搭建运行环境。

（6）安全加固：进一步完善接口限流、XSS 防护、CSRF 保护等安全机制，对敏感数据进行更严格的加密存储和传输保护。
