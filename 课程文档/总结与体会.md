## 2、实践体会

### 2.1 技术层面的收获

通过本次"基于知识图谱的软件工程课程问答平台的完整开发，我们深入实践了多项前沿技术的工程化应用。在后端方面，我们采用 Go 语言和 Gin 框架搭建了分层架构的 RESTful API 服务，深刻体会到 MVC 分层设计在大型项目中对代码可维护性和团队协作效率的提升作用。在前端方面，Vue 3 的 Composition API 和 TypeScript 的类型系统让复杂组件的状态管理更加清晰可控，D3.js 的力导向图算法为知识图谱的可视化提供了直观的交互体验。

在 AI 技术集成方面，我们首次实践了 RAG（检索增强生成）架构的完整流程：从文档解析、知识抽取、向量化存储到语义检索和大模型回答生成。Ollama 本地部署的大语言模型让我们对 LLM 的推理能力边界有了更直观的认识，也理解了向量检索在实际工程中的重要价值。Neo4j 图数据库的使用则让我们体会到了图结构在表示复杂知识关联时的独特优势，相比传统关系型数据库，图数据库在多跳查询和路径发现场景下具有显著的性能和表达力优势。

### 2.2 软件工程方法的实践

本次项目是软件工程课程知识的一次综合实践。我们严格按照软件开发生命周期开展工作：从需求分析出发，明确了学生和教师两类用户角色的功能需求；通过可行性分析论证了技术方案的合理性；在总体设计阶段完成了系统架构和模块划分；在详细设计阶段定义了数据模型和接口规范；最后通过黑盒测试验证了系统的功能完整性。

在需求分析过程中，我们学会了区分功能需求与非功能需求，理解了 RBAC 权限模型在多角色系统中的应用。在系统设计阶段，前后端分离的架构决策让我们认识到接口契约的重要性——清晰的 API 文档和统一的响应格式（code / message / data）极大减少了前后端联调的沟通成本。测试阶段的用例设计让我们体会到，测试不仅仅是"找 Bug"，更是对需求完整性和系统健壮性的系统性验证。

### 2.3 团队协作的经验

作为三人团队，我们采用了按技术栈分工的协作模式：后端、前端、AI 与测试文档各自独立推进，通过接口文档进行前后端联调。这种分工方式在项目初期效率较高，但也暴露了一些问题——例如 AI 模块的接口变更需要同步通知后端和前端，缺乏统一的接口版本管理机制导致了几次联调中的兼容性问题。

我们深刻认识到，在真实的企业级开发中，仅靠口头沟通和临时协调是远远不够的。未来应当引入更规范的协作流程，如 Git 分支管理策略（Git Flow）、Code Review 机制、CI/CD 流水线等，以提升团队协作的规范性和代码质量。

### 2.4 不足与改进方向

客观审视本项目，仍存在以下不足：

- **AI 模块的成熟度有限**：知识抽取依赖规则匹配和本地小参数模型，抽取准确率和覆盖面有待提升，未来可引入更强大的 NER/RE 模型或接入云端大模型 API
- **系统扩展性考虑不足**：当前为本地化单机部署，缺少容器化（Docker）和微服务拆分，难以支撑多用户并发场景
- **测试覆盖不够全面**：目前以黑盒测试为主，缺少单元测试和集成测试，代码质量保障体系不完善
- **用户体验细节待优化**：前端交互细节、错误提示友好度、响应式布局等方面还有改进空间

尽管存在不足，但本次项目让我们完整经历了一个软件产品从 0 到 1 的开发过程，将课堂上学到的理论知识真正落地为可运行的系统。这段经历不仅提升了我们的技术能力，更培养了工程化思维和团队协作意识，为今后的学习和工作奠定了坚实的基础。
