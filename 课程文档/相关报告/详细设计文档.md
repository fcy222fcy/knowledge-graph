1 引言
1.1 编写目的
本文档在需求分析和可行性分析基础上，对“基于知识图谱的软件工程课程问答平台”的软件结构、模块划分、数据库、知识图谱、向量检索、接口、关键流程、部署与测试方案进行详细设计，为后续编码实现、系统联调、测试验收和答辩演示提供依据。
本详细设计重点解决三个问题：第一，如何利用 Neo4j 将软件工程课程知识点组织为图结构；第二，如何利用本地大模型辅助抽取知识点、关系和生成答案；第三，如何通过向量检索与图谱查询结合，实现面向软件工程课程的可靠问答。
1.2 项目范围
本系统面向软件工程课程学习场景，主要服务学生、教师和管理员三类用户。学生可以进行自然语言提问、查看相关知识图谱和问答历史；教师可以上传课程资料、审核 AI 抽取结果、维护知识点和关系；管理员负责用户、角色、系统配置和运行监控。
本阶段采用本地化部署与增量迭代策略，优先保证课程资料上传、知识图谱构建、图谱可视化、语义问答和审核入库等核心功能闭环。题库生成、学习效果分析和个性化推荐作为增强模块进行设计，其中题库与学习分析可以按项目进度选择实现深度。

2 总体设计
2.1 设计目标
• 知识图谱真实落地：使用 Neo4j 存储审核通过的软件工程课程知识点和关系，而不是将知识点写死在代码中。
• AI 辅助构建：利用本地 Ollama qwen3:8b 从课程资料中生成候选知识点、候选关系和答案草稿，人工审核后入库。
• 混合检索问答：结合向量检索召回课程资料片段与 Neo4j 图谱扩展上下文，再由大模型组织答案。
• Go 工程化实现：以 Go Gin 作为主后端，统一负责业务接口、权限控制、任务调度、数据库访问、AI 调用和图谱操作。
• 可控可验收：为 AI 抽取结果设置待审核机制，为答案提供引用来源和相关知识点，降低模型幻觉风险。
2.2 总体架构
系统采用前后端分离架构。前端使用 Vue 3 + Element Plus 提供交互界面；后端使用 Go Gin 提供 RESTful API，并通过 GORM 访问 MySQL；MySQL 存储业务数据；Redis 负责异步任务、进度缓存和热点问答缓存；Neo4j 存储知识图谱并支持图查询；Ollama 提供本地大模型能力；向量检索用于课程资料片段语义召回。

5 数据库详细设计
5.1 MySQL 数据库设计原则
• MySQL 保存业务数据和审核过程，不作为知识图谱主存储；Go 后端统一通过 GORM 进行 ORM 映射和 CRUD 操作。
• 所有业务表保留 created_at、updated_at，核心数据保留 status 状态字段。
• 文档、分块、候选实体、候选关系、问答记录均保留来源，保证答案可追溯。
• 重要操作写入 audit_logs，便于验收和调试。
• 正式知识点以 Neo4j 为主，MySQL 可保存同步映射和业务索引。
5.2 GORM 持久化设计
GORM 作为 Go 后端访问 MySQL 的 ORM 框架，主要用于业务表的对象关系映射、基础 CRUD、分页查询、软删除、事务控制和表结构迁移。Neo4j 图谱数据仍通过 Neo4j Go Driver 和 Cypher 操作，Redis 仍通过 go-redis 操作，三者职责边界清晰。
• 模型映射：internal/model 目录定义 User、Document、DocumentChunk、ExtractionJob、ExtractedEntity、ExtractedRelation、QASession、QAMessage 等 GORM 模型。
• 字段约束：通过 gorm tag 声明主键、自增、唯一索引、普通索引、字段长度、默认值和 JSON 类型。
• 软删除：用户、文档、问答会话等可恢复数据采用 gorm.DeletedAt 实现软删除；候选抽取结果通过 status 控制审核状态。
• 事务控制：审核通过并写入正式数据时，MySQL 状态更新与 Neo4j 写入采用业务补偿机制；MySQL 内部多表修改使用 db.Transaction 保证一致性。
• 自动迁移：开发环境可使用 AutoMigrate 快速创建表结构；正式提交时保留 SQL 迁移脚本，避免表结构不可控变更。
GORM 连接初始化由 internal/pkg/mysql 完成，配置项从 config.yaml 中读取。连接池参数按照本地开发规模设置，后续可根据并发量调整 MaxOpenConns、MaxIdleConns 和 ConnMaxLifetime。
func InitMySQL(dsn string) (*gorm.DB, error) {
    db, err := gorm.Open(mysql.Open(dsn), &gorm.Config{
        DisableForeignKeyConstraintWhenMigrating: true,
    })
    if err != nil {
        return nil, err
    }
    sqlDB, err := db.DB()
    if err != nil {
        return nil, err
    }
    sqlDB.SetMaxOpenConns(20)
    sqlDB.SetMaxIdleConns(10)
    return db, nil
}

Repository 层不得将 GORM 查询细节暴露给 Handler。分页、排序、条件过滤和事务均由 Repository 或 Service 统一封装，保证接口层简洁。
type DocumentRepository struct {
    db *gorm.DB
}

func (r *DocumentRepository) Page(ctx context.Context, page, size int, status string) ([]model.Document, int64, error) {
    var items []model.Document
    var total int64
    q := r.db.WithContext(ctx).Model(&model.Document{})
    if status != "" {
        q = q.Where("status = ?", status)
    }
    if err := q.Count(&total).Error; err != nil {
        return nil, 0, err
    }
    err := q.Order("created_at DESC").
        Offset((page - 1) * size).
        Limit(size).
        Find(&items).Error
    return items, total, err
}

func (r *DocumentRepository) UpdateStatus(ctx context.Context, id uint, status string, message string) error {
    return r.db.WithContext(ctx).Model(&model.Document{}).
        Where("id = ?", id).
        Updates(map[string]any{"status": status, "error_message": message}).Error
}

在审核通过候选实体或候选关系时，Service 层先使用 GORM 更新 MySQL 审核状态，再调用 Neo4j 写入图谱。若 Neo4j 写入失败，系统将记录错误日志并把候选数据标记为 sync_failed 或保留待重试状态，避免 MySQL 状态与图数据库状态长期不一致。

6 知识图谱与向量检索设计
6.1 Neo4j 图谱模型
Neo4j 保存审核通过后的正式知识图谱。为保证模型简洁和便于维护，第一版采用通用节点标签 KnowledgePoint 表示课程知识点，采用统一关系类型 RELATION，并通过关系属性 type 表示具体语义关系。
表 6-1 Neo4j 节点设计
标签	含义	主要属性
KnowledgePoint	知识点节点	name、type、definition、aliases、source_document_id、created_at、updated_at、embedding
Document	课程资料节点	document_id、title、file_type、created_at
Chunk	课程资料分块节点	chunk_id、content、title_path、document_id、embedding
表 6-2 Neo4j 关系设计
关系模式	说明
(:Document)-[:HAS_CHUNK]->(:Chunk)	表示文档包含文本分块。
(:Chunk)-[:MENTIONS]->(:KnowledgePoint)	表示文本片段提到了某个知识点。
(:KnowledgePoint)-[:RELATION {type}]->(:KnowledgePoint)	表示知识点之间的语义关系，type 可为 属于、包含、前置于、输出、用于、对比等。

6.3 Cypher 写入设计
实体写入采用 MERGE 保证同名知识点不重复。关系写入前需确保源节点和目标节点均存在；若目标节点不存在，系统可将关系保持待审核状态或自动创建缺少定义的占位节点，由教师后续补全。
MERGE (n:KnowledgePoint {name: $name})
SET n.type = $type,
    n.definition = $definition,
    n.aliases = $aliases,
    n.source_document_id = $source_document_id,
    n.updated_at = datetime();

MATCH (a:KnowledgePoint {name: $source})
MATCH (b:KnowledgePoint {name: $target})
MERGE (a)-[r:RELATION {type: $relation_type}]->(b)
SET r.evidence = $evidence,
    r.updated_at = datetime();

6.4 向量检索设计
向量检索用于解决自然语言问法与标准术语不一致的问题。系统默认将课程分块 Chunk 和知识点定义向量化，用户提问时先生成问题向量，再检索相似 Chunk 和 KnowledgePoint。第一版建议使用 Neo4j Vector Index 承担向量索引，减少额外服务；若后续必须单独体现 FAISS，可在 vector.Repository 接口下替换为 FAISS 服务。
表 6-4 向量对象设计
向量对象	输入文本	用途
文本分块向量	Chunk.content	召回与问题语义接近的课程片段
知识点向量	KnowledgePoint.name + definition	召回相关知识点，辅助图谱扩展
问题向量	用户自然语言问题	作为检索查询向量
向量检索结果按照相似度排序后取 TopK，默认 K=5。若同时检索 Chunk 与 KnowledgePoint，则先分别召回，再通过知识点关系扩展补充一跳邻居节点。最终上下文按照“直接命中片段优先、图谱邻居补充、重复内容去重”的规则传给大模型。

10 安全、异常与日志设计
10.1 安全设计
• 认证：除登录注册外所有接口默认需要 JWT；Token 过期后前端跳转登录。
• 授权：教师和管理员接口使用角色中间件限制访问，学生不能调用审核和写图谱接口。
• 密码：使用 bcrypt 或 Argon2 进行不可逆哈希，不保存明文密码。
• 上传：限制文件类型、文件大小和保存路径，禁止用户控制最终文件路径。
• 输入校验：所有查询参数、JSON 请求体和路径参数进行校验，防止 SQL 注入和异常 Cypher 查询。
• AI 安全：提示词中限制模型只能根据检索上下文回答；模型输出不得直接执行为 SQL 或 Cypher。

AI 抽取不准确	知识点遗漏、关系错误、定义编造	必须人工审核；保留 evidence；设置关系白名单。
本地模型响应慢	问答等待时间长	使用 Redis 缓存、限制 TopK、支持任务异步和前端加载提示。
图谱关系混乱	关系类型过多、节点重复	统一 RELATION 加 type 属性，按实体名称去重，并在审核时合并。
开发周期不足	模块做不完	优先 P0 闭环：上传、抽取、审核、图谱、问答；题库和学习分析降级。
向量检索实现复杂	Neo4j Vector 或 FAISS 集成耗时	第一版可先实现关键词+图谱检索，再接入向量检索；保留 vector.Repository 抽象。
13.2 待确认事项
• 向量检索最终实现：默认采用 Neo4j Vector Index；若老师要求 FAISS，需要增加轻量 Python FAISS 服务或采用可替代方案。
• 文档格式支持范围：第一版建议优先支持 txt、md、docx，PDF 作为后续扩展或使用简单文本提取。
• 题库生成和学习分析深度：建议作为 P1 模块，按时间决定是否完整实现。
• 是否需要学校统一登录或只使用系统本地账号：当前设计默认使用本地账号和 JWT。
• Ollama embedding 模型是否已安装：若未安装，向量检索可先使用图谱检索和关键词检索作为过渡。
