## 1. 软件开发生命周期

### 1.1 SDLC 六大阶段

```
┌─────────────────────────────────────────────────────────┐
│                    SDLC 阶段                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  需求分析 → 系统设计 → 编码实现 → 测试 → 部署 → 维护    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

| 阶段 | 目标 | 关键产出 | 知识点 |
|------|------|----------|--------|
| 需求分析 | 明确系统功能和约束 | SRS文档、用户故事 | 功能/非功能需求、SMART原则 |
| 系统设计 | 将需求转化为技术方案 | 架构设计、数据库设计 | SOLID原则、设计模式 |
| 编码实现 | 编写可执行代码 | 源代码、单元测试 | 编程范式、代码质量 |
| 测试 | 验证软件质量 | 测试报告、缺陷列表 | 测试金字塔、TDD |
| 部署 | 发布到生产环境 | 部署文档、运维手册 | 蓝绿部署、金丝雀发布 |
| 维护 | 持续稳定运行 | 维护记录、升级计划 | 四种维护类型 |

### 1.2 SDLC 模型对比

| 模型 | 特点 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 瀑布模型 | 线性顺序 | 流程清晰、文档完善 | 灵活性差、风险后期发现 | 需求稳定项目 |
| 迭代模型 | 多次迭代 | 风险早期识别、客户早期参与 | 需要持续客户参与 | 中大型项目 |
| 增量模型 | 渐进式交付 | 快速交付核心功能 | 集成风险 | 分阶段交付 |
| 螺旋模型 | 风险驱动 | 风险控制、适合大型项目 | 成本较高 | 复杂项目 |
| V模型 | 测试导向 | 测试活动早期规划 | 灵活性差 | 关键系统 |

### 1.3 现代SDLC实践

**DevOps：**
- 开发和运维协作
- 自动化一切
- 持续改进

**敏捷开发：**
- 个体和互动高于流程和工具
- 工作的软件高于详尽的文档
- 客户合作高于合同谈判
- 响应变化高于遵循计划

**CI/CD：**
- 持续集成：频繁提交、自动化构建
- 持续交付：随时可部署
- 持续部署：自动发布

## 2. 敏捷开发方法论

### 2.1 敏捷宣言核心

```
我们正在揭示更好的软件开发方法，通过这些实践我们能够：
（个体和互动） 高于 （流程和工具）
（工作的软件） 高于 （详尽的文档）
（客户合作）   高于 （合同谈判）
（响应变化）   高于 （遵循计划）
```

**十二条原则要点：**
1. 尽早持续交付有价值的软件
2. 欢迎需求变化
3. 频繁交付可工作的软件
4. 业务人员和开发人员每天一起工作
5. 围绕有积极性的个人构建项目
6. 面对面交谈是最有效的沟通方式
7. 可工作的软件是进度的首要度量标准
8. 可持续的开发速度
9. 持续关注技术卓越
10. 简洁是必要的
11. 最好的架构出自自组织团队
12. 团队定期反思如何变得更有效

### 2.2 Scrum框架

**角色：**
- **产品负责人（PO）**：最大化产品价值，管理产品待办列表
- **Scrum Master（SM）**：服务型领导，确保Scrum流程执行
- **开发团队（DT）**：跨职能、自组织，负责交付产品增量

**事件：**
- **Sprint**：固定长度迭代（1-4周）
- **Sprint计划会议**：确定Sprint目标和待办列表
- **每日站会**：15分钟，同步进展和障碍
- **Sprint评审会议**：演示产品增量，收集反馈
- **Sprint回顾会议**：反思过程，制定改进计划

**工件：**
- **产品待办列表**：有序的产品需求列表
- **Sprint待办列表**：Sprint期间要完成的工作
- **产品增量**：Sprint结束时交付的产品

**用户故事格式：**
```
作为 <用户角色>,
我想要 <功能>,
以便 <商业价值/目标>.
```

**INVEST原则：**
- Independent（独立的）
- Negotiable（可协商的）
- Valuable（有价值的）
- Estimable（可估算的）
- Small（小的）
- Testable（可测试的）

### 2.3 Kanban方法

**六大实践：**
1. 可视化工作流
2. 限制在制品（WIP）
3. 管理流动
4. 明确流程策略
5. 实施反馈循环
6. 协作改进

**核心指标：**
- **周期时间**：从工作开始到完成的时间
- **吞吐量**：单位时间内完成的工作项数量
- **累积流图**：显示各状态的工作项分布

**Kanban vs Scrum：**

| 特性 | Kanban | Scrum |
|------|--------|-------|
| 迭代 | 持续流动 | 固定Sprint |
| 角色 | 无特定角色 | PO, SM, DT |
| 工件 | 看板、限制 | 产品/Sprint待办列表 |
| 变更 | 随时可添加 | Sprint期间不变更 |

### 2.4 极限编程（XP）

**核心实践：**
- 结对编程
- 测试驱动开发（TDD）
- 重构
- 持续集成
- 小版本发布
- 现场客户
- 编码规范
- 集体代码所有权

**XP价值观：**
- 沟通
- 简单
- 反馈
- 勇气
- 尊重

## 3. 软件架构设计模式

### 3.1 架构模式

**分层架构：**
```
表现层 → 业务逻辑层 → 数据访问层 → 数据库层
```
- 优点：结构清晰、易于维护
- 缺点：性能开销、可能单体化

**MVC架构：**
```
模型(M) ←→ 视图(V) ←→ 控制器(C)
```
- 模型：数据和业务逻辑
- 视图：用户界面展示
- 控制器：处理用户输入

**微服务架构：**
```
API网关 → 服务A, 服务B, 服务C, 服务D
              ↓      ↓      ↓      ↓
           数据库A 数据库B 数据库C 数据库D
```
- 优点：技术异构、独立部署、故障隔离
- 缺点：分布式复杂性、数据一致性挑战

**事件驱动架构：**
```
事件生产者 → 事件总线 → 事件消费者
```
- 优点：松耦合、高可扩展性
- 缺点：调试困难、事务管理复杂

### 3.2 创建型设计模式

**单例模式：**
- 确保一个类只有一个实例
- 提供全局访问点
- 适用：数据库连接池、日志记录器

**工厂方法模式：**
- 定义创建对象的接口
- 让子类决定实例化哪个类
- 适用：不确定具体对象类型

**建造者模式：**
- 将复杂对象的构建与其表示分离
- 允许使用相同的构建过程创建不同的表示
- 适用：创建复杂对象

### 3.3 结构型设计模式

**适配器模式：**
- 将一个类的接口转换成客户期望的另一个接口
- 适用：集成遗留系统、使用第三方库

**外观模式：**
- 为子系统中的一组接口提供一个一致的接口
- 适用：简化复杂子系统

**代理模式：**
- 为其他对象提供一种代理以控制对这个对象的访问
- 适用：远程服务调用、延迟加载、访问控制

### 3.4 行为型设计模式

**策略模式：**
- 定义一系列算法，将每个算法封装起来
- 适用：算法需要在运行时切换

**观察者模式：**
- 定义对象间的一对多依赖关系
- 适用：事件处理系统、数据绑定

**命令模式：**
- 将请求封装为对象
- 适用：撤销/重做功能、事务处理

## 4. 软件测试策略

### 4.1 测试金字塔

```
          /\
         /  \        验收测试（少量）
        /----\
       /      \      系统测试（适量）
      /--------\
     /          \    集成测试（适量）
    /------------\
   /              \  单元测试（大量）
  /----------------\
```

**原则：**
- 越接近底层，测试数量越多
- 越接近底层，测试速度越快
- 越接近底层，测试成本越低

### 4.2 测试类型

**单元测试：**
- 测试单个组件是否按预期工作
- 目标：验证代码逻辑正确性
- 最佳实践：AAA模式（Arrange-Act-Assert）

**集成测试：**
- 测试多个组件一起工作时的行为
- 目标：验证组件间接口
- 策略：自顶向下、自底向上、大爆炸

**系统测试：**
- 测试完整的集成系统是否符合需求
- 目标：验证端到端功能
- 类型：功能测试、非功能测试

**验收测试：**
- 验证系统是否满足业务需求
- 目标：由客户或最终用户执行
- 类型：UAT、运营验收、合同验收

### 4.3 测试方法

**黑盒测试技术：**
- 等价类划分
- 边界值分析
- 决策表
- 状态转换测试

**白盒测试技术：**
- 语句覆盖
- 分支覆盖
- 条件覆盖
- 路径覆盖

**专项测试：**
- 性能测试：负载、压力、并发
- 安全测试：OWASP Top 10
- 可用性测试：用户测试、启发式评估

### 4.4 测试自动化

**测试金字塔自动化：**
- 单元测试：多，快速反馈
- 集成测试：适量，服务虚拟化
- 系统测试：适量，API自动化
- 验收测试：少，UI自动化

**测试驱动开发（TDD）：**
1. 编写失败的测试（红）
2. 编写最少代码使测试通过（绿）
3. 重构代码（重构）
4. 重复

**行为驱动开发（BDD）：**
```gherkin
Feature: 用户登录
  Scenario: 成功登录
    Given 用户在登录页面
    When 用户输入正确的用户名和密码
    Then 用户应该被重定向到仪表板
```

## 5. 版本控制与Git工作流

### 5.1 Git基础

**Git对象模型：**
- Blob：文件内容
- Tree：目录结构
- Commit：提交对象
- Tag：标签对象

**Git区域：**
```
工作区 → git add → 暂存区 → git commit → 本地仓库 → git push → 远程仓库
```

**常用命令：**
```bash
git init              # 初始化仓库
git clone <url>       # 克隆仓库
git status            # 查看状态
git add <file>        # 添加到暂存区
git commit -m "msg"   # 提交
git push              # 推送到远程
git pull              # 拉取远程更新
git log               # 查看历史
```

### 5.2 分支操作

```bash
git branch <name>     # 创建分支
git checkout -b <name>  # 创建并切换分支
git merge <branch>    # 合并分支
git rebase <branch>   # 变基
```

**合并 vs 变基：**
- 合并：保留完整历史，创建合并提交
- 变基：线性历史，不创建合并提交
- 原则：公共分支用合并，私有分支可用变基

### 5.3 Git工作流

**Git Flow：**
- master：生产环境代码
- develop：开发主线
- feature：功能开发
- release：发布准备
- hotfix：紧急修复

**GitHub Flow：**
- 只有master分支
- 基于Pull Request
- 持续部署

**选择原则：**
- 小型团队、简单项目：GitHub Flow
- 发布周期明确：Git Flow
- 持续部署：GitHub Flow

### 5.4 代码审查

**Pull Request流程：**
1. 创建Pull Request
2. 描述变更内容
3. 指定审查者
4. 进行代码审查
5. 处理审查反馈
6. 合并Pull Request

**审查检查项：**
- 代码质量：编码规范、代码异味
- 功能正确性：需求实现、边界处理
- 可维护性：可读性、测试覆盖
- 安全性：安全漏洞、敏感数据

### 5.5 提交信息规范

**格式：**
```
<类型>(<范围>): <描述>

<详细说明>

<关联Issue>
```

**类型：**
- feat：新功能
- fix：修复bug
- docs：文档更新
- refactor：重构
- test：测试相关
- chore：构建或工具相关

## 6. DevOps与CI/CD

### 6.1 DevOps核心

**DevOps = Development + Operations**

**四大原则：**
1. 人和过程优先于工具
2. 自动化一切
3. 持续改进
4. 度量驱动

**DevOps生命周期：**
```
计划 → 编码 → 构建 → 测试 → 发布 → 部署 → 运维 → 监控
  ↑                                                    ↓
  ←←←←←←←←←←←← 反馈循环 ←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←
```

### 6.2 持续集成（CI）

**核心实践：**
- 频繁提交代码（每天多次）
- 每次提交都触发自动化构建
- 自动化测试验证
- 快速反馈问题

**CI工作流程：**
```
代码提交 → 触发构建 → 编译 → 单元测试 → 代码质量检查 → 生成报告
```

**CI工具：**
- Jenkins、GitLab CI/CD、GitHub Actions、CircleCI

### 6.3 持续交付/部署（CD）

**持续交付：**
- 代码可以随时部署到生产环境
- 部署需要手动批准
- 降低发布风险

**持续部署：**
- 代码通过所有测试后自动部署到生产环境
- 无需人工干预
- 快速反馈循环

**部署策略：**
- 蓝绿部署：两个完全相同的生产环境
- 金丝雀发布：逐步向用户推送更新
- 滚动部署：逐个更新服务器

### 6.4 基础设施即代码（IaC）

**核心理念：**
- 基础设施用代码定义
- 版本控制基础设施
- 自动化基础设施管理

**工具：**
- Terraform：多云支持、声明式配置
- Ansible：无代理架构、YAML配置
- Docker：容器化技术
- Kubernetes：容器编排

### 6.5 监控与可观测性

**三大支柱：**
- 指标（Metrics）：数值型数据
- 日志（Logs）：事件记录
- 追踪（Traces）：请求链路

**监控工具：**
- Prometheus + Grafana：指标监控
- ELK Stack：日志管理
- Jaeger：分布式追踪

**DORA指标：**
1. 部署频率
2. 变更前置时间
3. 变更失败率
4. 故障恢复时间

## 7. 核心概念速查

### 7.1 设计原则

| 原则 | 描述 | 关键点 |
|------|------|--------|
| SOLID | 面向对象设计原则 | 单一职责、开闭、里氏替换、接口隔离、依赖倒置 |
| DRY | 不要重复自己 | 消除重复代码 |
| KISS | 保持简单愚蠢 | 简单设计 |
| YAGNI | 你不会需要它 | 避免过度设计 |

### 7.2 常见模式

| 类别 | 模式 | 适用场景 |
|------|------|----------|
| 创建型 | 工厂方法 | 创建对象 |
| 创建型 | 单例 | 全局唯一实例 |
| 结构型 | 适配器 | 接口转换 |
| 结构型 | 外观 | 简化接口 |
| 行为型 | 策略 | 算法切换 |
| 行为型 | 观察者 | 事件通知 |

### 7.3 测试概念

| 概念 | 描述 | 关键点 |
|------|------|--------|
| 测试金字塔 | 测试层次结构 | 单元测试多，验收测试少 |
| TDD | 测试驱动开发 | 红-绿-重构 |
| BDD | 行为驱动开发 | Gherkin语法 |
| 代码覆盖率 | 测试覆盖程度 | 语句、分支、路径覆盖 |

### 7.4 Git概念

| 概念 | 描述 | 关键点 |
|------|------|--------|
| 分支 | 并行开发线 | 轻量级、快速切换 |
| 合并 | 集成分支变更 | 保留历史、解决冲突 |
| 变基 | 重写提交历史 | 线性历史、私有分支 |
| Pull Request | 代码审查流程 | 协作、质量保证 |

### 7.5 DevOps概念

| 概念 | 描述 | 关键点 |
|------|------|--------|
| CI | 持续集成 | 频繁提交、自动化构建 |
| CD | 持续交付/部署 | 随时可发布、自动发布 |
| IaC | 基础设施即代码 | 版本控制、自动化管理 |
| 监控 | 可观测性 | 指标、日志、追踪 |

## 8. 最佳实践总结

### 8.1 开发最佳实践

1. **代码质量**
   - 遵循编码规范
   - 进行代码审查
   - 保持简单设计
   - 持续重构

2. **测试实践**
   - 测试驱动开发
   - 自动化测试
   - 测试金字塔
   - 持续测试

3. **版本控制**
   - 频繁提交
   - 小步提交
   - 清晰的提交信息
   - 分支管理

### 8.2 架构最佳实践

1. **设计原则**
   - SOLID原则
   - 关注点分离
   - 松耦合高内聚
   - 保持简单

2. **模式选择**
   - 根据问题选择模式
   - 避免过度设计
   - 考虑扩展性
   - 权衡取舍

### 8.3 运维最佳实践

1. **自动化**
   - 构建自动化
   - 测试自动化
   - 部署自动化
   - 监控自动化

2. **监控**
   - 全方位监控

## 10. 总结

软件工程是一个不断发展的领域，核心知识体系包括：

1. **生命周期管理**：从需求到维护的完整流程
2. **方法论选择**：敏捷、Scrum、Kanban等
3. **架构设计**：模式选择、设计原则
4. **质量保证**：测试策略、自动化测试
5. **版本控制**：Git工作流、代码审查
6. **DevOps实践**：CI/CD、自动化、监控

关键是要理解这些知识的内在联系，根据实际情况灵活应用，持续学习和改进。软件工程不仅是技术，更是协作、沟通和持续改进的艺术。
