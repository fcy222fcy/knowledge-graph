## 1. 软件开发生命周期（SDLC）

### 1.1 SDLC 定义

软件开发生命周期（Software Development Life Cycle, SDLC）是一种结构化的软件开发过程框架，定义了从需求收集到软件维护的完整流程。它提供了一套系统化的方法来规划、创建、测试和部署高质量的软件产品。

### 1.2 SDLC 六大阶段

```
┌─────────────────────────────────────────────────────────┐
│                    SDLC 阶段                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  需求分析 → 系统设计 → 编码实现 → 测试 → 部署 → 维护    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

| 阶段 | 目标 | 关键产出 | 知识点 |
|------|------|----------|--------|
| **需求分析** | 明确系统功能和约束 | SRS文档、用户故事 | 功能/非功能需求、SMART原则 |
| **系统设计** | 将需求转化为技术方案 | 架构设计、数据库设计 | SOLID原则、设计模式 |
| **编码实现** | 编写可执行代码 | 源代码、单元测试 | 编程范式、代码质量 |
| **测试** | 验证软件质量 | 测试报告、缺陷列表 | 测试金字塔、TDD |
| **部署** | 发布到生产环境 | 部署文档、运维手册 | 蓝绿部署、金丝雀发布 |
| **维护** | 持续稳定运行 | 维护记录、升级计划 | 四种维护类型 |

### 1.3 需求分析阶段详解

**主要活动：**
- 与利益相关者（Stakeholders）沟通，收集需求
- 分析现有系统和业务流程
- 识别功能需求和非功能需求
- 创建需求规格说明书（SRS）

**关键产出：**
- 需求规格说明书（SRS）
- 用户故事（User Stories）
- 用例图（Use Case Diagrams）
- 需求追踪矩阵

**最佳实践：**
- 使用可测量的需求描述
- 建立需求优先级（MoSCoW法则）
- 进行需求评审和确认
- 管理需求变更

**MoSCoW法则：**
- **Must have**：必须实现的核心功能
- **Should have**：应该实现的重要功能
- **Could have**：可以实现的次要功能
- **Won't have**：本次不实现的功能

### 1.4 系统设计阶段详解

**主要活动：**
- 架构设计（高阶设计）
- 详细设计（低阶设计）
- 数据库设计
- 接口设计
- 安全设计

**关键产出：**
- 架构设计文档
- 数据库ER图
- API接口文档
- 安全设计文档

### 1.5 编码实现阶段详解

**主要活动：**
- 编写代码实现功能
- 代码评审
- 单元测试
- 集成测试

**最佳实践：**
- 遵循编码规范
- 进行代码评审
- 编写单元测试
- 持续集成

### 1.6 测试阶段详解

**测试金字塔：**
```
          /\
         /  \        验收测试 (少量)
        /----\
       /      \      系统测试 (适量)
      /--------\
     /          \    集成测试 (适量)
    /------------\
   /              \  单元测试 (大量)
  /----------------\
```

**测试类型：**
- **单元测试**：测试单个函数或方法
- **集成测试**：测试模块之间的交互
- **系统测试**：测试整个系统
- **验收测试**：验证是否满足需求

### 1.7 部署阶段详解

**部署策略：**
- **蓝绿部署**：维护两套环境，切换流量
- **金丝雀发布**：逐步扩大用户范围
- **滚动部署**：逐个更新实例
- **A/B测试**：对比不同版本效果

### 1.8 维护阶段详解

**维护类型：**
- **纠正性维护**：修复缺陷
- **适应性维护**：适应环境变化
- **完善性维护**：增加新功能
- **预防性维护**：预防潜在问题

### 1.9 SDLC 模型对比

| 模型 | 特点 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **瀑布模型** | 线性顺序 | 流程清晰、文档完善 | 灵活性差、风险后期发现 | 需求稳定项目 |
| **迭代模型** | 多次迭代 | 风险早期识别、客户早期参与 | 需要持续客户参与 | 中大型项目 |
| **增量模型** | 渐进式交付 | 快速交付核心功能 | 集成风险 | 分阶段交付 |
| **螺旋模型** | 风险驱动 | 风险控制、适合大型项目 | 成本较高 | 复杂项目 |
| **V模型** | 验证与确认对应 | 测试早期规划 | 灵活性差 | 安全关键系统 |
| **敏捷模型** | 迭代增量 | 快速响应变化、客户参与 | 需要高度协作 | 需求变化快 |

## 2. 敏捷开发方法论

### 2.1 敏捷宣言

```
我们正在揭示更好的软件开发方法，通过这些实践我们能够：
（个体和互动） 高于 （流程和工具）
（工作的软件） 高于 （详尽的文档）
（客户合作）   高于 （合同谈判）
（响应变化）   高于 （遵循计划）
```

**注意：** 敏捷宣言并不是说右侧不重要，而是说左侧更重要。

### 2.2 敏捷十二条原则

1. **最高优先级：** 通过尽早和持续地交付有价值的软件来满足客户
2. **欢迎变化：** 即使在开发后期也欢迎需求变化
3. **频繁交付：** 经常交付可工作的软件，交付周期越短越好
4. **协作：** 业务人员和开发人员必须每天一起工作
5. **激励个体：** 围绕有积极性的个人构建项目
6. **面对面交谈：** 面对面交谈是最有效的沟通方式
7. **可工作软件：** 可工作的软件是进度的首要度量标准
8. **可持续：** 提倡可持续的开发速度
9. **技术卓越：** 持续关注技术卓越和良好设计
10. **简洁：** 最大化未完成工作量的艺术是必要的
11. **自组织：** 最好的架构、需求和设计出自自组织团队
12. **定期反思：** 团队定期反思如何变得更有效

### 2.3 Scrum 框架

**Scrum 角色：**
- **产品负责人（Product Owner）**：管理产品待办列表，确定优先级
- **Scrum Master**：确保Scrum流程执行，移除障碍
- **开发团队（Development Team）**：跨职能自组织团队

**Scrum 事件：**
- **Sprint**：固定时长的迭代（通常2-4周）
- **Sprint计划会**：规划本Sprint的工作
- **每日站会**：每日15分钟同步进度
- **Sprint评审会**：展示完成的工作
- **Sprint回顾会**：反思改进

**Scrum 产出：**
- **产品待办列表（Product Backlog）**：需求列表
- **Sprint待办列表（Sprint Backlog）**：本Sprint任务
- **增量（Increment）**：可交付的产品增量

### 2.4 Kanban 看板

**核心概念：**
- **可视化工作流**：使用看板展示任务状态
- **限制在制品（WIP）**：限制同时进行的任务数量
- **管理流动**：优化任务流转效率
- **明确流程策略**：定义工作规则
- **持续改进**：定期优化流程

**看板列：**
```
待办 | 进行中 | 代码评审 | 测试 | 完成
```

### 2.5 极限编程（XP）

**核心实践：**
- **结对编程**：两人共同编码
- **测试驱动开发（TDD）**：先写测试再写代码
- **持续集成**：频繁集成代码
- **重构**：持续改进代码结构
- **简单设计**：只做当前需要的
- **集体代码所有权**：团队共同维护代码
- **现场客户**：客户全程参与
- **编码规范**：统一编码风格

## 3. 软件架构设计模式

### 3.1 分层架构（Layered Architecture）

**描述：**
系统被组织成多个水平层，每层只与相邻层交互。

**典型分层：**
```
┌─────────────────────────────────┐
│        表现层 (Presentation)     │  ← 用户界面、API
├─────────────────────────────────┤
│        业务逻辑层 (Business)     │  ← 核心业务规则
├─────────────────────────────────┤
│        数据访问层 (Data Access)   │  ← 数据库操作
├─────────────────────────────────┤
│        数据库层 (Database)        │  ← 数据存储
└─────────────────────────────────┘
```

**优点：**
- 结构清晰，易于理解
- 关注点分离
- 易于维护和测试
- 可以独立替换某一层

**缺点：**
- 性能开销（层间调用）
- 可能导致单体应用
- 更新可能需要跨层
- 灵活性有限

**适用场景：**
- 传统企业应用
- 简单的Web应用
- 团队经验不足时

### 3.2 MVC 架构（Model-View-Controller）

**描述：**
将应用分为三个核心组件：模型、视图和控制器。

**组件职责：**
- **Model（模型）**：数据和业务逻辑
- **View（视图）**：用户界面展示
- **Controller（控制器）**：处理用户输入，协调Model和View

**优点：**
- 关注点分离
- 易于扩展
- 支持多种视图
- 便于单元测试

**适用场景：**
- Web应用
- 移动应用
- 桌面应用

### 3.3 微服务架构（Microservices）

**描述：**
将应用拆分为一组小型、自治的服务，每个服务负责特定业务功能。

**特点：**
- 服务独立部署
- 服务独立扩展
- 技术异构性
- 去中心化治理

**优点：**
- 灵活性高
- 可扩展性强
- 技术栈灵活
- 故障隔离

**缺点：**
- 分布式系统复杂性
- 网络通信开销
- 数据一致性挑战
- 运维复杂度高

**适用场景：**
- 大型复杂系统
- 需要独立部署
- 团队规模大
- 高并发场景

### 3.4 事件驱动架构（Event-Driven Architecture）

**描述：**
基于事件的发布-订阅模式，组件通过事件进行通信。

**核心概念：**
- **事件（Event）**：系统中发生的事情
- **事件生产者**：发布事件的组件
- **事件消费者**：订阅并处理事件的组件
- **事件通道**：传递事件的媒介

**优点：**
- 松耦合
- 异步处理
- 可扩展性强
- 实时性好

**适用场景：**
- 实时数据处理
- 用户行为追踪
- 系统解耦
- 流式处理

### 3.5 管道-过滤器架构（Pipe-and-Filter）

**描述：**
数据通过一系列处理步骤（过滤器）流动，每个步骤处理后传递给下一步。

**特点：**
- 每个过滤器独立
- 过滤器可复用
- 支持并行处理
- 易于扩展

**适用场景：**
- 数据处理流水线
- 编译器设计
- 图像处理
- ETL流程

## 4. 软件测试策略

### 4.1 测试金字塔

```
          /\
         /  \        验收测试 (少量)
        /----\
       /      \      系统测试 (适量)
      /--------\
     /          \    集成测试 (适量)
    /------------\
   /              \  单元测试 (大量)
  /----------------\
```

**原则：**
- 越接近底层，测试数量越多
- 越接近底层，测试速度越快
- 越接近底层，测试成本越低
- 越接近底层，测试维护成本越低

### 4.2 测试类型

**按测试对象分类：**
- **功能测试**：验证软件功能是否符合需求
- **非功能测试**：验证性能、安全、可用性等
- **结构测试**：基于代码结构的测试

**按测试方法分类：**
- **黑盒测试**：不考虑内部实现，只关注输入输出
- **白盒测试**：基于代码结构和逻辑
- **灰盒测试**：结合黑盒和白盒

### 4.3 测试原则

1. **测试显示缺陷存在**：测试不能证明没有缺陷
2. **穷尽测试是不可能的**：使用风险分析和测试技术
3. **测试活动必须尽早开始**：缺陷发现越早，修复成本越低
4. **缺陷聚集**：大部分缺陷集中在少部分模块
5. **杀虫剂悖论**：重复相同的测试会失效
6. **测试是上下文相关的**：不同系统需要不同测试方法

### 4.4 单元测试

**定义：**
测试单个函数、方法或类的正确性。

**最佳实践：**
- 每个测试只验证一个行为
- 测试命名清晰（如：`TestCalculateTotal_WithValidItems_ReturnsSum`）
- 使用Arrange-Act-Assert模式
- 测试边界条件和异常情况
- 保持测试独立性

**TDD（测试驱动开发）流程：**
```
1. Red：编写一个失败的测试
2. Green：编写最少代码使测试通过
3. Refactor：重构代码保持测试通过
```

### 4.5 集成测试

**定义：**
测试多个模块或服务之间的交互。

**策略：**
- **自顶向下**：从顶层模块开始，使用桩模块模拟下层
- **自底向上**：从底层模块开始，使用驱动模块测试上层
- **大爆炸**：所有模块一起测试
- **三明治**：结合自顶向下和自底向上

### 4.6 系统测试

**定义：**
测试完整的系统是否满足需求。

**类型：**
- **功能测试**：验证功能正确性
- **性能测试**：验证响应时间、吞吐量
- **安全测试**：验证安全性
- **兼容性测试**：验证跨平台兼容性
- **用户验收测试**：用户验证是否满足需求

### 4.7 测试自动化

**自动化测试金字塔：**
- **单元测试**：最容易自动化，速度最快
- **API测试**：较容易自动化，速度适中
- **UI测试**：较难自动化，速度较慢

**工具选择：**
- **单元测试**：JUnit、pytest、Jest
- **API测试**：Postman、REST Assured
- **UI测试**：Selenium、Cypress、Playwright

## 5. DevOps与持续集成持续部署

### 5.1 DevOps 定义

**DevOps = Development + Operations**

**核心理念：**
- **文化**：打破开发和运维的壁垒，建立协作文化
- **自动化**：自动化一切可以自动化的流程
- **度量**：通过数据驱动决策和改进
- **共享**：知识、责任和成功共享

**DevOps目标：**
- 更快的交付速度
- 更高的发布频率
- 更低的故障率
- 更快的故障恢复
- 更好的协作效率

### 5.2 DevOps 原则

1. **人和过程优先于工具**
2. **自动化一切**
3. **持续改进**
4. **度量驱动**

### 5.3 持续集成（CI）

**定义：**
开发人员频繁地将代码集成到共享仓库，每次集成都通过自动化构建和测试验证。

**核心实践：**
- 维护单一代码仓库
- 自动化构建
- 自动化测试
- 快速构建
- 测试环境与生产环境一致
- 每次提交都触发构建

**CI 流程：**
```
代码提交 → 自动构建 → 自动测试 → 生成报告 → 反馈
```

### 5.4 持续交付/部署（CD）

**持续交付（Continuous Delivery）：**
确保代码随时可以部署到生产环境，但部署需要手动触发。

**持续部署（Continuous Deployment）：**
自动将通过所有测试的代码部署到生产环境。

**CD 流程：**
```
代码提交 → 自动构建 → 自动测试 → 自动部署到 staging → 手动批准 → 部署到生产
```

### 5.5 CI/CD 工具链

**版本控制：**
- Git、GitHub、GitLab、Bitbucket

**构建工具：**
- Jenkins、GitLab CI、GitHub Actions、Travis CI

**容器化：**
- Docker、Podman

**编排：**
- Kubernetes、Docker Swarm

**配置管理：**
- Ansible、Chef、Puppet

**监控：**
- Prometheus、Grafana、ELK Stack

### 5.6 DevOps 工具链全景

```
计划 → 编码 → 构建 → 测试 → 发布 → 部署 → 运维 → 监控
  ↓      ↓      ↓      ↓      ↓      ↓      ↓      ↓
Jira   Git   Jenkins  JUnit  Artifactory  K8s   Ansible  Prometheus
```

## 6. 设计原则与模式

### 6.1 SOLID 原则

| 原则 | 全称 | 含义 |
|------|------|------|
| **SRP** | 单一职责原则 | 一个类只有一个引起它变化的原因 |
| **OCP** | 开闭原则 | 对扩展开放，对修改关闭 |
| **LSP** | 里氏替换原则 | 子类必须能够替换父类 |
| **ISP** | 接口隔离原则 | 客户端不应该依赖它不需要的接口 |
| **DIP** | 依赖倒置原则 | 依赖抽象而非具体实现 |

### 6.2 其他设计原则

- **DRY（Don't Repeat Yourself）**：避免代码重复
- **KISS（Keep It Simple, Stupid）**：保持简单
- **YAGNI（You Aren't Gonna Need It）**：不过度设计
- **关注点分离**：不同功能分开实现
- **组合优于继承**：优先使用组合而非继承

### 6.3 创建型模式

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **单例模式** | 确保一个类只有一个实例 | 全局配置、日志记录 |
| **工厂模式** | 定义创建对象的接口 | 对象创建逻辑复杂 |
| **抽象工厂** | 创建相关对象的家族 | 跨平台UI组件 |
| **建造者模式** | 分步构建复杂对象 | 对象有多个可选参数 |
| **原型模式** | 通过复制创建对象 | 创建成本高的对象 |

### 6.4 结构型模式

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **适配器模式** | 将一个类的接口转换为客户期望的接口 | 集成第三方库 |
| **桥接模式** | 将抽象与实现分离 | 跨平台开发 |
| **组合模式** | 将对象组合成树形结构 | 文件系统、UI组件树 |
| **装饰器模式** | 动态添加职责 | 不修改对象添加功能 |
| **外观模式** | 为子系统提供统一接口 | 简化复杂系统 |
| **享元模式** | 共享细粒度对象 | 文本编辑器字符对象 |
| **代理模式** | 为对象提供代理以控制访问 | 远程代理、虚拟代理 |

### 6.5 行为型模式

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **观察者模式** | 定义对象间一对多依赖 | 事件系统、消息队列 |
| **策略模式** | 定义算法族，使它们可互换 | 支付方式、排序算法 |
| **模板方法** | 定义算法骨架，子类重写具体步骤 | 框架设计 |
| **迭代器模式** | 顺序访问聚合对象元素 | 集合遍历 |
| **命令模式** | 将请求封装为对象 | 撤销/重做、宏命令 |
| **状态模式** | 允许对象在状态改变时改变行为 | 订单状态、TCP连接 |
| **责任链模式** | 使多个对象都有机会处理请求 | 审批流程、异常处理 |

## 7. 版本控制与Git工作流

### 7.1 Git 基础

**核心概念：**
- **仓库（Repository）**：存储项目代码和历史
- **提交（Commit）**：代码变更的快照
- **分支（Branch）**：独立的开发线
- **合并（Merge）**：合并分支变更
- **远程（Remote）**：远程仓库

**基本工作流：**
```
工作区 → 暂存区 → 本地仓库 → 远程仓库
(git add)  (git commit)  (git push)
```

### 7.2 Git 分支策略

**Git Flow：**
```
main ───────────────────────────────────────────
  │                               ↑
  └── develop ────────────────────┼────────────
       │        ↑                 │
       └── feature/xxx ──────────┘
            │
            └── release/1.0 ─────┘
                 │
                 └── hotfix/xxx
```

**分支类型：**
- **main**：生产环境代码
- **develop**：开发主线
- **feature/**：新功能开发
- **release/**：发布准备
- **hotfix/**：紧急修复

### 7.3 GitHub Flow

**流程：**
1. 从main分支创建feature分支
2. 在feature分支开发
3. 创建Pull Request
4. 代码评审
5. 合并到main
6. 部署

### 7.4 GitLab Flow

**环境分支策略：**
```
main → production
     → staging
     → development
```

### 7.5 提交规范

**Commit Message 格式：**
```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 类型：**
- **feat**：新功能
- **fix**：修复bug
- **docs**：文档更新
- **style**：代码格式（不影响功能）
- **refactor**：重构
- **test**：测试相关
- **chore**：构建/工具相关

## 8. 项目管理与质量保证

### 8.1 项目管理知识领域

1. **范围管理**：定义和控制项目范围
2. **时间管理**：规划和控制项目进度
3. **成本管理**：规划和控制项目预算
4. **质量管理**：规划和保证项目质量
5. **人力资源管理**：组建和管理项目团队
6. **沟通管理**：规划和管理项目沟通
7. **风险管理**：识别和应对项目风险
8. **采购管理**：管理项目采购
9. **干系人管理**：管理项目干系人期望

### 8.2 质量保证活动

**代码评审：**
- 检查代码质量
- 发现潜在问题
- 分享知识
- 统一编码风格

**静态代码分析：**
- 代码复杂度分析
- 代码重复检测
- 安全漏洞扫描
- 代码规范检查

**度量指标：**
- **代码覆盖率**：测试覆盖的代码比例
- **缺陷密度**：每千行代码的缺陷数
- **技术债务**：需要修复的技术问题
- **代码复杂度**：圈复杂度等指标

### 8.3 风险管理

**风险管理流程：**
1. **风险识别**：识别潜在风险
2. **风险分析**：评估概率和影响
3. **风险应对**：制定应对策略
4. **风险监控**：持续监控风险

**风险应对策略：**
- **规避**：消除风险
- **转移**：将风险转移给第三方
- **减轻**：降低风险概率或影响
- **接受**：接受风险
